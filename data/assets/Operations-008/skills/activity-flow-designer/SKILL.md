# 活动流程设计 Skill

帮助运营人员系统化地完成活动流程设计与文档输出工作，确保活动方案完整、可执行、规范化。

> ⚠️ **本节是 Skill 的第一个执行动作，优先级高于一切其他内容。读取本文件后，必须立即输出以下声明，不得跳过、不得延后、不得在声明前输出任何其他内容。**

## 一、Skill 目的

本 Skill 用于指导 AI 完成运营活动流程设计与文档输出工作，核心任务如下：

1. **需求分析与参数收集** — 收集活动基本信息，锁定场景类型与对应规则文件
2. **方案设计** — 基于用户需求与行业规则模板，系统化产出完整的活动方案（含12维规则体系、流程节点设计等）
3. **文档生成** — 将设计方案整合为结构化的活动流程执行方案文档

### 适用范围

适用于各类运营活动场景（线上 / 线下 / 混合），包括但不限于：促销交易、用户增长、品牌传播、会员运营、节日大促等。

### 错误处理机制（结构化错误码）

当 Skill 遇到异常时，必须返回 `[错误码] + 一句话摘要 + 处置动作`，禁止直接崩溃或输出空泛内容。

| 错误码 | 异常情况 | 处置措施 | 重试/回退策略 |
|-------|---------|----------|-------------|
| `E001` | 必填参数空值 | 暂停输出，列出缺失项清单，引导用户补充 | 可无限重试，直到必填参数齐全 |
| `E002` | 活动类型无法判定 | 列出可选类型供选择；仍无法判定则请求参考案例 | 重试1次，仍无法判定则按「未分类」继续并标注 |
| `E003` | 规则文件读取失败 | 降级为通用SOP标准继续执行，末尾标注待补充项 | 不重试，直接降级回退 |
| `E004` | 用户中途变更需求 | 确认变更范围，保留已确认的有效参数，仅对变更部分重新收集 | 增量重试，仅重新执行受影响的部分 |
| `E005` | 部分维度信息不足 | 先按通用规则生成草案，标注「待确认」并列出需补充问题 | 以草案交付，附带待确认清单 |
| `E006` | 输出超出长度限制 | 按模块分步输出，等待确认后再继续下一部分 | 自动分片，每片独立确认后继续 |

> 示例：`[E001] 缺少必填参数：活动名称、活动类型。请补充以下信息后再继续...`

### 多轮对话引导原则

1. **渐进式收集** — 优先收集最关键的2-3个参数（名称→类型→目标），其余边做边确认
2. **提供选项而非开放式提问** — 当用户不确定时，给出可选项，降低回答成本
3. **用草案换信息** — 参数无法获取时先用合理默认值生成草案，请用户审阅时顺便补全
4. **每次追问只问一件事** — 避免连续多轮问答造成疲劳，每轮对话至少产出一部分有价值的内容

## 三、规则要求

### 3.1 方案设计阶段

> 本阶段基于用户输入的需求参数与行业规则模板，系统化产出完整的活动方案设计。

#### 3.1.1 需求理解

**活动背景收集：**
- [ ] 活动目的与目标
- [ ] 目标用户群体画像与分层
- [ ] 活动时间周期与关键时间点
- [ ] 预算范围与成本结构
- [ ] 资源约束条件（技术、人力、渠道、场地等）
- [ ] 核心规则雏形（参与规则、激励机制、准入限制）

**核心规则识别：**
- [ ] 参与者参与规则雏形
- [ ] 激励/激励机制框架
- [ ] 时间节点规则概要
- [ ] 准入条件与限制

#### 3.1.2 规则设计维度（12个）

> 以下维度的具体表述需根据活动实际形态灵活调整。例如：线上活动称"用户/入口/库存"，线下活动称"参与者/通道/物料/名额"。

**参与与任务规则：**

| 维度 | 设计要点 |
|-----|---------|
| 参与资格 | 准入条件、次数限制 |
| 参与渠道 | 触达路径与进入方式 |
| 操作流程 | 参与动作与步骤 |
| 任务规则 | 完成判定标准 |

**激励与资源规则：**

| 维度 | 设计要点 |
|-----|---------|
| 激励内容 | 奖励/权益/体验/荣誉（按形态选用） |
| 激励兑现 | 交付方式、有效期、使用范围 |
| 存量管理 | 有限资源的总量管控与消耗追踪 |
| 资源配置 | 人力/物料/场地/资金/系统的规划调配 |

**体验与风控规则：**

| 维度 | 设计要点 |
|-----|---------|
| 展示与触达 | 信息内容与触达方式 |
| 分享与互动 | 传播规范与互动规则 |
| 风控策略 | 作弊防范、风险控制与安全保障 |
| 异常与收尾 | 异常处置方案与收尾安排 |

#### 3.1.3 流程节点设计

> 节点是通用的抽象框架，具体表述需根据活动形态（线上/线下/混合）灵活适配。

**活动生命周期：**

| 阶段 | 核心任务 | 设计要点 |
|-----|---------|---------|
| 筹备期 | 资源准备、配置确认、预热宣发 | 准入条件、测试/彩排验证、资源到位检查 |
| 预热期 | 用户蓄水、信息触达、规则预热 | 预热激励、预约/报名规则、名额管理 |
| 正式期 | 核心活动流程执行 | 完整参与规则、过程管控、动态调整 |
| 收尾期 | 激励交付、结果统计、参与者通知 | 延迟交付规则、最终口径、收尾事项 |
| 复盘期 | 效果评估、经验沉淀 | 数据归档、复盘框架、案例沉淀 |

**关键节点（按活动运行自然顺序组织）：**

*启停与阶段节点：*

| 节点 | 说明 | 设计方向 |
|------|------|--------|
| 活动启动 | 正式面向参与者开放 | 启动时间、检查清单、开放方式 |
| 阶段转换 | 内部环节/回合的切换 | 阶段定义、触发条件、衔接安排 |
| 活动结束 | 正式关闭，不再接受新参与 | 结束时间、结束方式、善后安排 |

*参与链路节点：*

| 节点 | 说明 | 设计方向 |
|------|------|--------|
| 参与核验 | 确认参与者是否符合资格 | 核验标准、核验方式、不通过处理 |
| 任务达成 | 完成规定的动作或目标 | 达成判定标准、确认方式、即时反馈 |
| 激励资格判定 | 确认是否获得激励及内容 | 判定逻辑、匹配规则、特殊情况处理 |

*激励链路节点：*

| 节点 | 说明 | 设计方向 |
|------|------|--------|
| 激励交付 | 激励转移到可支配状态 | 交付时机、交付方式、失败重试/替补 |
| 激励核销/使用 | 实际消耗或兑现所获激励 | 使用条件、核销方式、有效期管理 |
| 激励回退 | 已交付激励收回或失效 | 回退触发条件、已用激励处理、补偿方案 |

*资源与异常节点：*

| 节点 | 说明 | 设计方向 |
|------|------|--------|
| 资源存量变化 | 关键资源的消耗或补充 | 监控阈值、补充机制、耗尽应对 |
| 异常处置 | 过程中偏离预期的情况 | 异常分级、响应策略、升级路径、恢复流程 |
| 数据归集截止 | 统计数据的时间界限 | 截止时点定义、延迟数据归属规则 |

#### 3.1.4 动态规则引用

根据 `{scene_id}` 读取 `references/` 目录下对应规则文件，严格遵循其中的：

1. **行业评分标准** — 评估该场景下方案设计的完整性与质量
2. **专属违禁词库** — 该场景下禁止使用的表述或规则

## 四、工作流

```
开始
  ↓
【步骤1】初始化
  ├─ 输入：用户的原始需求描述
  ├─ 处理：收集必填参数（名称→类型→目标），锁定 scene_id 与 rule_file
  ├─ 输出：参数集（内存/上下文）
  ├─ 步骤完成时必须输出：「【步骤1 完成】参数集已锁定：activity_name=X / activity_type=X / scene_id=X / rule_file=X，进入步骤2」
  └─ 出口条件：✅activity_name / ✅activity_type / ✅activity_goal 三项齐备
  ↓
【步骤2】方案设计
  ├─ 输入：步骤1的参数集
  ├─ 处理：需求理解 → 类型锁定(读rules) → 12维规则设计 → 节点设计
  ├─ 中间产物：方案设计草稿
  ├─ 步骤完成时必须输出：「【步骤2 完成】12维规则设计已产出，进入步骤3」
  └─ 出口条件：12个维度均有产出
  ↓
【步骤3】文档生成
  ├─ 输入：步骤1参数集 + 步骤2方案设计草稿
  ├─ 处理：按方案设计内容组装 → 生成输出产物
  ├─ 步骤完成时必须输出：「【步骤3 完成】文档已生成」
  └─ 出口条件：输出符合用户 instruction 定义的格式要求
  ↓
结束 → 向用户返回文件路径列表 + 一句话交付说明
```

### 状态传递规范

1. **跨步骤传递摘要** — 后续步骤通过摘要获取前序产物的关键信息，避免大段文本占用上下文窗口
2. **摘要内容约定**：
   - 步骤2→3摘要：参与链路核心逻辑、激励链路核心逻辑、Top3风控点、需要用户确认的待定项列表
