# 数据可视化规则

## 核心原则

**用户指定优先，未指定时 Agent 自主推断。不含数据分析本身，仅负责将分析结果转化为图表。**

本环节接收数据文件（可能来自「数据分析」环节的输出，也可能由用户直接提供）。

### 语言规范

可视化输出中的所有文字内容（报告标题、章节标题、图表标题、坐标轴标签、图例、注释等），**默认使用与用户指令相同的语言**。

| 用户指令语言 | 输出语言 | 说明 |
|------------|---------|------|
| 中文 | 中文 | 标题、标签、图例、注释全部使用中文 |
| 英文 | 英文 | 全部使用英文 |
| 未明确（纯数据文件无文字指令） | 中文 | 默认使用中文 |

> **双语字体适配：英文输出时字体回退链中英文字体（如 Arial、Helvetica）优先级可适当提升，但必须保留中文字体作为兜底**（因为图表数据标签可能仍包含中文内容）。中文输出时保持现有中文优先回退链不变。

---

## 0. 默认结构与格式规范（用户未指定可视化要求时必遵循）

当用户未指定具体的可视化结构、样式或排版要求时，**必须遵循以下默认规范**生成报告。这些规范旨在保证输出的一致性、可读性及跨平台兼容性，避免出现文字显示异常（如黑色方框/乱码/方块字）等问题。

### 0.0 输出模式判定

在开始生成任何内容之前，必须先根据用户指令判断输出模式：

| 判定条件 | 输出模式 | 行为说明 |
|---------|---------|---------|
| 用户明确提到「报告」、「report」、「文档」等词汇 | **报告模式** | 生成完整报告（含标题体系、章节结构、分析文字），按 §0.1~§0.5 的报告规范执行；输出格式按 §0.6 多格式规范处理 |
| 用户仅提到「画图」、「图表」、「可视化」、「plot」、「chart」等，**未提及**「报告」 | **图表模式** | 仅生成独立图表图片，不生成报告文档；按 §0.2' 图表模式专用规范 执行 |
| 用户完全未指定 / 仅提供数据文件要求分析 | **图表模式**（默认） | 同上，仅输出图表图片 |
| 用户同时指定了输出格式（如「输出PDF」、「导出Word」） | **报告模式** | 按用户指定的格式生成报告 |

> **关键规则：除非用户明确要求「报告」，否则默认走「图表模式」，只输出图片文件到文件夹，不拼接报告文档。**

### 0.1 报告层次结构（仅报告模式适用）

所有可视化报告（HTML / PDF / Word / 图片组合报告）必须采用 **3 层标题体系**：

| 层级 | 用途 | 格式要求 | 中文示例 | 英文示例 |
|------|------|---------|---------|----------|
| **H1 — 报告标题** | 整份报告的名称，唯一 | 使用数据集/业务主题命名，简洁明确 | `Q1 渠道流量效果分析报告` | `Q1 Channel Traffic Analysis Report` |
| **H2 — 章节标题** | 按分析模块分章节，对应一类图表 | 使用「模块名 + 分析角度」格式 | `一、渠道流量概览`、`二、转化效果分析` | `1. Channel Traffic Overview`、`2. Conversion Analysis` |
| **H3 — 图表标题** | 每张图表的独立标题，一一对应 | 统一使用「图表编号 + 描述 + 图表类型」格式 | `图1：各渠道UV占比（饼图）` | `Figure 1: UV Share by Channel (Pie Chart)` |

**规则说明：**
- H1 仅出现 1 次（报告开头）
- H2 按分析模块数量动态生成，通常 3~6 个章节
- H3 与每张图表严格一一对应，不允许有多图共用一个标题或无标题图表
- 编号规则随语言自适应：
  - **中文**：H2 用「一、二、三、…」，H3 用「图1、图2、图3、…」
  - **英文**：H2 用「1. 2. 3. …」，H3 用「Figure 1, Figure 2, Figure 3, …」

### 0.2 图表命名规范

每张图表的文件名和标题必须遵循统一格式：

```
文件名格式：{序号}_{描述文本}_.{扩展名}

中文示例：01_各渠道UV占比_饼图.png 、02_日PV趋势变化_折线图.png 、03_渠道CTR对比_柱状图.png
英文示例：01_uv_share_by_channel_pie.png 、02_daily_pv_trend_line.png 、03_ctr_comparison_bar.png

图表内标题格式（跟随输出语言）：
  中文 → 图{序号}：{描述}（{图表类型}）     示例：图1：各渠道UV占比（饼图）
  英文 → Figure {N}: {Description} ({Type})   示例：Figure 1: UV Share by Channel (Pie Chart)
```

**要求：**
- 文件名中的**描述文本使用与输出语言一致的字符**（中文输出用中文描述、英文输出用英文描述）
- 文件名中**禁止包含空格和特殊符号**（如 `\ / : * ? " < > |`），使用下划线 `_` 替代空格
- 描述文本应能概括图表核心内容，控制在 3~8 个字符（中文）或 3~5 个单词（英文）以内
- 图表内标题放在图表上方居中位置，语言与用户指令一致

### 0.3 字体规范（关键：防止文字变黑框/方块）

字体选择是避免「文字显示为黑色方框/乱码」的核心手段。**必须严格遵守以下字体策略：**

#### 字体回退链（Fallback Chain）

在代码中设置字体时，**禁止只指定单一字体**，必须按以下优先级配置完整的字体回退链。Agent 自主编写具体实现代码。

| 用途 | 字体族 | 回退优先级（从高到低） |
|------|--------|---------------------|
| **标题 / 正文 / 图表内文字**（默认） | sans-serif | Microsoft YaHei → SimHei → PingFang SC → Hiragino Sans GB → WenQuanYi Micro Hei → Noto Sans CJK SC → Arial Unicode MS → DejaVu Sans → Arial → sans-serif |
| **数值标签 / 刻度 / 等宽场景** | monospace | Consolas → Monaco → Courier New → monospace |

> 以上回退链同时适用于：matplotlib 绑定、HTML 报告 CSS `font-family`、PDF/Word 中文字体选择。英文输出时可适当将英文字体（Arial/Helvetica）优先级前移，但中文字体必须保留在链中。

#### 分级字体大小规范

| 元素 | 字号（推荐） | 最小字号 | 说明 |
|------|------------|---------|------|
| 报告标题 H1 | 20~24pt | 18pt | 加粗 |
| 章节标题 H2 | 16~18pt | 14pt | 加粗 |
| 图表标题 H3 | 13~15pt | 12pt | 加粗 |
| 坐标轴标签 | 11~12pt | 10pt | 常规 |
| 图例文字 | 10~11pt | 9pt | 常规 |
| 注释/标注文字 | 9~10pt | 8pt | 常规 |

> **警告：任何文字元素字号不得小于 8pt**，过小字号在某些渲染引擎下会退化为方块。

#### 中文字体显式注册（matplotlib 场景）

当使用 matplotlib 绑定中文字体时，**必须在绘图前执行字体注册**。Agent 自主编写代码，采用以下两种方案之一：

| 方案 | 适用场景 | 操作要点 |
|------|---------|---------|
| **A — FontProperties 显式指定**（首选） | 单个文本元素精确控制 | 通过 `FontProperties(family="字体名", size=字号)` 创建对象，传给 `fontproperties=` 参数。最可靠，不受全局配置影响 |
| **B — rcParams 全局配置**（备选） | 批量统一所有图表字体 | 设置 `rcParams["font.sans-serif"]` 为上述回退链列表；**必须同时设置 `rcParams["axes.unicode_minus"] = False`**，否则负号可能显示为方块 |

> 无论使用哪种方案，绘图前都应扫描系统可用字体（`fontManager.ttflist`），优先选用系统中实际存在的字体。

#### 禁止事项

| 禁止操作 | 原因 | 替代方案 |
|---------|------|---------|
| 只写 `fontname="SimHei"` 不加 fallback | 非 Windows 系统可能无该字体 | 使用完整回退链 |
| 使用 `fontname="STSong"` 等罕见字体 | 大部分系统未预装 | 使用通用中文字体 |
| 在 HTML 报告中使用 `font-family: '宋体'` 单一声明 | macOS/Linux 无宋体 | 使用 `"Microsoft YaHei", "PingFang SC", sans-serif` 回退链 |
| 字号小于 8pt | 渲染引擎可能退化显示 | 最小 8pt，推荐 ≥9pt |
| 不设置 `axes.unicode_minus = False` | 负号 "-" 可能显示为方块 | 显式设置为 False |
| 硬编码绝对字体文件路径 | 其他机器上路径不存在 | 使用字体名称或动态查找 |

### 0.2' 图表模式专用规范（仅图表模式适用）

当输出模式为「图表模式」时，遵循以下简化规范：

#### 输出内容
- **仅生成独立的图表图片文件**，不生成任何报告文档（HTML/PDF/Word 均不生成）
- 每张图表包含：图表内标题（遵循 §0.2 的标题格式）、坐标轴标签、图例、数据标注（如有必要）
- 不需要 H1/H2 章节结构，但每张图仍需有 H3 级别的图表内标题

#### 默认保存路径与格式
| 项目 | 默认值 | 可被用户覆盖 |
|------|--------|------------|
| **保存目录** | `结果/图表/` （在输出根目录下创建 `图表` 子文件夹） | 用户指定路径时按用户路径 |
| **默认格式** | `PNG` | 用户可指定为 SVG / PDF / JPG 等 |
| **DPI** | 200（独立交付的高清图片） | — |
| **尺寸** | 宽 12inch × 高 8inch（横幅）或 10×10（方形，根据图表类型自适应） | — |

> 图表模式下生成的每张图片仍需执行 §3 的质量检查环节。

### 0.4 配色规范

- 使用专业配色方案（如 `tab10`、`Set2`、`Blues` 等色阶），避免高饱和度荧光色
- 同一报告中相同维度/类别颜色保持一致
- 数据系列数量 ≤ 8 时使用 `tab10` 或 `Set2`；> 8 时使用 `tab20` 或渐变色阶
- 背景统一使用白色或浅灰（`#FAFAFA`），禁止深色背景（打印/导出时易出问题）

### 0.5 图表尺寸与布局

| 输出形式 | 推荐尺寸 | DPI |
|---------|---------|-----|
| 单张图片（嵌入报告用） | 宽 10inch × 高 6inch（横幅）或 8×8（方形） | 150 |
| 单张图片（独立交付 / 图表模式） | 宽 12inch × 高 8inch | 200 |
| HTML 报告内嵌图 | 宽 100%（响应式高度自适应） | — |
| 多子图组合 | 根据子图数量自动计算，保持每个子图不小于 4×3 inch | 150 |

### 0.6 多格式输出规范（仅报告模式适用）

当输出模式为「报告模式」时，支持以下输出格式：

| 格式 | 触发条件 | 默认选择 | 说明 |
|------|---------|---------|------|
| **HTML** | 未指定格式时的默认格式 | ✅ 默认 | 自包含的 HTML 文件（内嵌 CSS/图表 Base64 或引用图片），可在浏览器中直接打开 |
| **PNG 图片集** | 用户指定「图片」/「PNG」/「JPG」 | — | 每张图独立一个 PNG 文件 + 可选的汇总索引页 |
| **PDF** | 用户指定「PDF」/「pdf」 | — | 使用图表生成 PDF 报告，保留完整的标题体系和排版 |
| **Word (.docx)** | 用户指定「Word」/「docx」/「Word文档」 | — | 使用 python-docx 生成 Word 报告，嵌入图表图片 |

#### 各格式的特殊要求

**HTML 报告：**
- 单文件自包含（CSS 内联、图表可 Base64 内嵌或相对路径引用）
- 字体回退链通过 `<style>` 注入，确保离线打开中文正常显示
- 响应式设计，支持不同屏幕尺寸浏览

**PDF 报告：**
- 使用 matplotlib 的 `PdfPages` 或 reportlab 生成
- 中文字体必须嵌入 PDF（`fontproperties` 指定物理字体文件路径或使用已注册字体）
- 页面大小默认 A4 纵向，图表跨页时自动分页
- 每页包含页眉（报告标题）和页码

**Word 报告：**
- 使用 python-docx 库生成
- 按 H1 → H2 → H3 层次设置 Word 标题样式（标题1/标题2/标题3）
- 图表以嵌入图片方式插入（PNG 格式，DPI=200），居中显示
- 自动生成目录页（TOC）
- 正文使用宋体/等线（Windows）或 PingFang SC（macOS），字号 11pt

> **格式优先级：用户显式指定 > 默认 HTML > 按输出场景推断。**

## 1. 常规可视化

**无需规则，模型自主探索数据并根据数据特征自行决定绘制哪些图表。**

> 选择原则：一图一结论，不画无信息量的图。
> **必须遵守第 0 节的所有默认结构与格式规范。注意先按 §0.0 判定输出模式，再决定是否执行 §0.1 报告层次结构。**

## 2. 用户分析可视化（用户未指定时）

**根据输入数据中包含的分析结果或数据特征，遍历可绘制项，应绘制的都绘制。**

输入数据可能来自「用户分析」环节的输出，也可能是用户直接提供的已有分析文件。

> **必须遵守第 0 节的所有默认结构与格式规范。注意先按 §0.0 判定输出模式，再决定是否执行 §0.1 报告层次结构。**

### 可绘制清单

遍历以下与输入数据内容匹配的绘制项：

| 用户分析模块 | 可绘制内容 |
|-------------|-----------|
| 用户画像 | 各属性维度的分布图、价值分布直方图、行为频次分布 |
| RFM 分析 | RFM 分组分布柱状图、RFM 两两维度交叉散点图、各分组用户数对比 |
| 留存分析 | Cohort 留存热力图、留存衰减折线图、不同分组留存曲线对比 |
| 流失与复购 | 流漏斗图、复购次数分布、流失 vs 未流失用户特征对比图 |
| 用户分群 | 各群体规模占比饼图、群体间核心指标对比雷达图 |
| 生命周期 | 阶段分布漏斗图/桑基图、阶段流转路径 |
| 趋势与对比（有 时间字段时） | 时间序列折线图（多指标）、环比/同比柱状图或组合图（本期 vs 上期叠加展示）、同比环比变化率趋势线 |

- 某绘制项因数据不足无法生成时跳过，在输出摘要中说明原因
- **不限于上表**，Agent 可根据具体分析结果补充有价值的可视化角度

## 3. 可视化输出检查环节（必执行）

**在生成所有可视化图表并组装报告后、交付给用户之前，必须执行以下检查流程。** 本环节用于发现和修复常见的渲染问题，尤其是「文字显示为黑色方框/方块/乱码」这一类问题。

**无论图表模式还是报告模式，均须执行本环节。** 图表模式仅检查图片文件；报告模式还需额外检查生成的报告文件。

### 3.1 检查时机

```
图表模式：所有图表生成完毕 → 【执行图片检查】 → 修复问题（如有）→ 重新导出 → 交付用户
报告模式：所有图表生成完毕 → 组装报告 → 【执行全量检查（图片+报告文件）】 → 修复问题（如有）→ 重新导出 → 交付用户
```

### 3.2 检查任务清单

生成可视化后，**必须按以下任务逐项执行检查**。具体脚本的编写和实现由 Agent 自主完成。

#### 总体流程

遍历所有生成的输出文件，根据文件扩展名分派对应的检查任务：

| 文件类型 | 需执行的检查任务 |
|---------|---------------|
| PNG / JPG / JPEG / SVG / TIFF | 图片质量检查（C01~C05） |
| PDF | 报告文件完整性检查（C08~C09）+ 编码/乱码检查（C14） |
| DOCX / DOC | 报告文件完整性检查（C10~C12）+ 编码/乱码检查（C15） |
| HTML / HTM | 报告文件完整性检查（C06~C07）+ 编码/乱码检查（C13） |

---

#### 任务一：图片质量检查（对应 C01~C05）

对每张生成的图片依次执行以下 5 项检查：

| 步骤 | 检查项 | 具体要求 | 判定标准 | 修复方式 |
|------|--------|---------|---------|---------|
| 1 | **文件完整性** | 验证图片能否正常解码打开 | 无法打开 → C02 致命 | 排查代码逻辑 |
| 2 | **尺寸检查** | 检查宽高是否低于最小阈值（400×300px） | 低于阈值 → C04 警告 | 调大尺寸重渲染 |
| 3 | **黑色方框/方块字检测**（核心） | 检测图片是否存在因字体缺失导致的文字变黑框/方块现象。采用双重策略：<br>① **像素级分析**：统计近黑像素（RGB 均 <30）占比，超过 15% 时触发进一步验证；<br>② **形态学验证**：对近黑区域做连通域分析，筛选填充率 >0.82 且尺寸在 8~80px 的规则矩形块，可疑数量 ≥3 则判定为方块字；<br>③ **纹理辅助**：将图上半区域（标题区）切分为小窗口，计算各窗口标准差，平坦暗区窗口占比 >30% 则预警 | 检出方块字 → C01 致命 | 更换字体回退链配置后重新渲染所有图表 |
| 4 | **空白图像检测** | 计算全图像素标准差，判断是否接近纯色（全白/全黑） | 标准差 <3 → C03 致命 | 排查数据和绑图代码 |
| 5 | **标签溢出检测** | 检查图片右边缘 5% 区域是否存在非背景色内容 | 存在且非坐标轴 → C05 警告 | 尝试调整布局边距 |

> **方块字修复方案：** 扫描系统可用中文字体 → 强制更新 matplotlib rcParams（`font.sans-serif` + `axes.unicode_minus=False`）→ 重新渲染所有图表 → 再次检查。

---

#### 任务二：报告文件完整性检查（对应 C06~C12）

**HTML 文件（C06~C07）：**
- 内容长度检查（<500 字符视为异常）
- 基本 HTML 结构校验（必须包含 `<html>` / `<body>` / `<head>` 标签）
- CSS `font-family` 是否包含中文字体回退链（无则警告）
- 图片引用路径有效性检查（`src` / `href` 引用的图片文件必须存在）

**PDF 文件（C08~C09）：**
- 页数检查（页数为 0 视为生成失败）
- 逐页文本提取，判断空白页数量（文本长度 <10 视为空页）

**Word 文件（C10~C12）：**
- 有效段落数量检查（无内容段落视为异常）
- 内嵌图片数量检查（可视化报告应有嵌入图片）
- 文件大小检查（<1KB 视为内容不完整）

---

#### 任务三：编码与乱码检查（对应 C13~C15）

**C13 — HTML 编码异常/乱码检测：**
- **替换字符检测**：扫描 U+FFFD 替换字符数量，>0 即判定为编码异常
- **Mojibake 模式匹配**：检测典型的乱码特征序列（如 `Ã¤Â¸­`、`锟斤拷`、`浣犲ソ` 等），出现 ≥3 次即判定为乱码
- **Charset 声明校验**：检查 `<meta charset>` 声明的编码是否为 UTF-8，非 UTF-8 则警告
- **UTF-8 解码失败时的自动修复**：若以 UTF-8 打开文件抛出 `UnicodeDecodeError`，自动尝试 GBK → GB2312 → GB18030 → Latin-1 → Big5 逐个探测原始编码，成功后以 UTF-8 重写文件

**C14 — PDF 文本乱码/不可读检测：**
- 提取 PDF 第一页文本样本（需 >20 字符）
- 计算「可读字符率」=（中文 + 英文 + 数字 + 常见标点）÷ 总字符数
- 可读率 <40% → 判定为乱码疑似（字体未嵌入）；可读率 40%~70% → 警告

**C15 — Word 内容乱码检测：**
- 收集文档全部正文 + 表格文本
- 扫描替换字符（U+FFFD）→ >0 即编码异常
- 扫描典型乱码标志（`锟斤拷` / `浣犲ソ` / `Ã`）→ 出现 ≥2 次即乱码疑似
- **乱码文本修复**：多策略尝试还原被错误解码的文本——依次尝试 UTF-8↔GBK↔Latin-1↔GB2312 的编解码组合，通过对比修复前后乱码标志字符数量的下降幅度来验证修复效果

### 3.4 问题处置流程

```
执行检查
  ├─ 全部通过 → 直接交付用户
  └─ 存在问题
       ├─ 存在 🔴 致命问题
       │    ├─ C01（方块字）      → 更新字体回退链配置 → 重新渲染所有图表 → 再次检查
       │    ├─ C02/C03（图片损坏/空白）→ 排查数据和绑图代码 → 重新生成该图表
       │    ├─ C07（HTML引用缺失）→ 修正路径引用 → 重新导出
       │    ├─ C08/C10/C12（报告文件异常）→ 排查报告生成代码 → 重新生成报告
       │    ├─ C13（HTML编码异常）   → 自动探测原始编码并转 UTF-8 重写 → 重新导出
       │    ├─ C14（PDF文本乱码）   → 重新生成 PDF 并确保字体物理路径嵌入 → 再次检查
       │    ├─ C15（Word内容乱码）  → 多策略修复源数据乱码 → 重新生成 Word → 再次检查
       │    └─ 重试后仍失败       → 向用户报告错误详情和建议解决方案，不交付有问题的文件
       └─ 仅存在 🟡 警告问题
            → 记录到 warnings 列表
            → 尝试自动修复：
            │   C04 调大尺寸 / C05 调整边距 / C06 注入 CSS
            │   C09/C11 建议人工确认
            → 交付，并在输出摘要中附上「质量检查备注」列出所有警告
```

### 3.5 交付标准

**只有满足以下条件时才可将可视化报告交付给用户：**

1. ✅ 无 🔴 致命级别问题（C01~C05 中任意 + C07/C08/C10/C12/C13/C14/C15 中任意）
2. ✅ 所有图表文字清晰可读，无方块字/乱码
3. ✅ 所有图片文件可正常打开
4. ✅ 报告文件（HTML/PDF/Word）可正常打开且内容完整
5. ✅ HTML 报告中图片引用路径有效（如有）
6. ⚠️ 允许存在 🟡 警告级别问题（C04~C06、C09、C11、C14低可读率警告），但必须在输出摘要中注明
