# 数据汇总规则

## 核心原则

**用户指定优先，未指定时 Agent 自主推断。**

以下仅列出需要 Agent 遵守的约束规则。数据读取、字段匹配、合并执行等基础能力由 Agent 自主完成。

---

## 1. 用户可指定的参数

| 参数 | 说明 | 未指定时的默认行为 |
|------|------|-------------------|
| `文件` | 参与合并的文件 | 当前目录下全部数据文件 |
| `Sheet` | 使用的 Sheet | 全部 |
| `字段` | 使用的字段（包含/排除） | 全部字段 |
| `筛选条件` | 数据行过滤条件（如日期范围、渠道等） | 不过滤 |
| `关联键` | 合并关联字段 | 自动探测共同维度字段，探测后需确认 |
| `合并模式` | `join` / `append` | 根据数据源结构自动判断 |
| `主数据源` | 多源 join 时作为基础表（Left Table）的数据源 | 按以下优先级确定：①用户明确指定 → ②用户输入列表中的第一个 → ③当用户指定的是文件夹（而非具体文件）时，取文件夹内行数最多的文件；未指定且无法判断时需确认 |
| `列对齐策略`（仅 append 模式有效） | `并集`（保留所有列，缺失填 NaN） / `交集`（只保留共同拥有的列） | 默认取**并集** |
| `字段映射` | 字段间的对应关系 | 智能判断同义字段并统一命名 |
| `输出字段名` | 输出列的命名规则 | 取最具代表性的名称 |
| `输出格式` | `xlsx` / `csv` | 单一输入→与输入一致；多输入→`xlsx` |
| `输出方式` | 单文件 / 按某维度拆分为多文件（如按渠道/平台/日期各一个文件） | 默认输出单个汇总文件 |
| `单元格格式` | 数值/日期等单元格的显示格式（小数位数、千分位、百分比、日期样式等） | 与输入文件保持一致；多输入格式不统一时以第一个文件为基准 |

## 2. 合并执行规则

### 2.0 前置检查

在执行任何合并操作之前，必须按顺序完成以下前置检查：

| 检查项 | 处理方式 |
|--------|---------|
| **编码/字符集检测** | 逐文件检测编码格式（UTF-8/GBK/GB2312/UTF-16 等），发现不一致时统一转为 UTF-8；若某文件读取时出现乱码或解码失败，尝试备选编码重试，仍失败则暂停并向用户确认 |
| **数据量评估** | 估算合并后总行数、总列数与总内存占用（行×列×单单元格预估大小）。触发分块处理的阈值：**（a）总行数 >100万** 或 **（b）预估内存 >500MB** 或 **（c）总行数 >50万且列数 >100**（宽表低行数场景同样可能压力大）。命中任一条件即启用分块处理策略：逐文件分块读取 → 分块合并 →逐步写入输出文件，避免一次性全量加载导致内存溢出 |
| **时间范围校验**（多源 join 时） | 检查各数据源的时间字段范围。当任意两个数据源的时间重叠部分不足任一方时间跨度的 **50%** 时，**必须暂停并向用户确认**是否继续（对应 §3「时间范围严重不重叠」确认场景），不可静默跳过 |

### 2.1 append 模式的列对齐规则

当 `合并模式` 为 `append`（纵向拼接）时，不同数据源的列名可能不完全一致，按以下规则处理：

| 场景 | 处理方式 |
|------|---------|
| 所有数据源列完全一致 | 直接拼接，无需额外处理 |
| 列名不一致但存在同义字段 | 先按「字段映射」规则将同义字段统一命名后再拼接 |
| 列对齐策略为 `并集`（默认） | 保留所有出现过的列，某数据源缺少该列时对应位置填充空值（NaN），并在问题清单中记录哪些列来源于哪些数据源 |
| 列对齐策略为 `交集` | 仅保留所有数据源共同拥有的列，独有列被丢弃并在问题清单中记录 |
| 用户指定了 `列对齐策略` | 严格按用户指定执行 |

> **注意**：append 模式下列不对齐是高频场景，Agent 应主动检查列一致性而非静默忽略。

### 2.2 多源 join 的连接顺序规则

当 2 个及以上数据源以 `join` 模式合并时，必须遵循以下连接顺序逻辑：

1. **确定主数据源**：按以下优先级确定 Left Table：**①用户明确指定了主数据源 → ②用户输入列表中的第一个文件 → ③当用户指定的是文件夹（而非具体文件列表）时，取该文件夹内行数最多的文件**。**2 个数据源 join 时同样适用本规则——主数据源为 Left Table，另一个为 Right Table**
2. **确定连接链路**：以主数据源为基础，将其余数据源逐个 Left Join 进来。当数据源≥3 个时，连接顺序按**关联键匹配度从高到低**排列（即与主数据源共同字段最多的数据源优先连接）
3. **连接过程中的冲突**：每一步 join 产生的冲突按 §3（冲突处理规则）的三层优先级处置
4. **无法自动确定顺序时**：当多个数据源与主数据源的关联键匹配度相同或均无可探测关联键时，暂停并向用户建议连接顺序

> **设计原则**：确保主数据源的行不丢失（Left Join 语义），避免因 Inner Join 导致数据莫名减少。

### 2.3 冲突处理规则

#### 2.3.1 处置优先级（三层判定）

遇到冲突时，按以下顺序决定处置方式，**高优先级覆盖低优先级**：

| 优先级 | 判定条件 | 处置依据 |
|-------|---------|---------|
| **P1 — 用户已明确指定** | 用户指令中已说明该类冲突如何处理（如"数值不一致取来源A"、"重复行直接去重"） | **严格按用户指令执行** |
| **P2 — 触发确认场景** | 用户未指定，且该冲突属于 §3「必须确认的场景」范围 | **暂停并向用户确认**，不得自作主张 |
| **P3 — 默认处置** | 用户未指定，且不属于必须确认的场景 | **按下表默认规则执行** |

> 核心逻辑：先回溯用户指令是否已澄清 → 未澄清则判断是否需要问 → 都不需要才走默认处置。

#### 2.3.2 默认处置规则（P3 兜底）

仅当 P1、P2 均不命中时，适用以下默认规则：

| 冲突类型 | 默认处理方式 |
|---------|-------------|
| 同一字段在不同数据源中数值不一致（含差异>20%但经用户确认继续合并的情况） | **保留多版本**，分别命名为 `[字段]_来源A`、`[字段]_来源B`，标注警告；差异超过 20% 时额外标注「⚠️ 显著差异」标记 |
| 某数据源有而其他没有的字段 | 缺失处填空值（NaN），**不插值不猜测** |
| 同一数据源内存在重复行（相同关联键） | **暂不去重**，记录数量并提示，留至清洗环节 |
| 关联键在某数据源中有空值 | 该行无法匹配时保留为独立行（类似 Left Join），并标注 |
| 同一字段在不同数据源中数据类型不一致（如文本 vs 日期、带千分位 vs 浮点数） | **自动对齐为兼容类型**（文本→统一转为最宽松的数值/日期类型；带千分位/单位符号的数值→去除格式符后转浮点数），对齐失败则保留多版本并标注警告 |

## 3. 必须确认的场景

以下情况**必须暂停并向用户确认**，其余场景直接处理：

| 场景 | 触发条件 | 用户可如何避免被询问 |
|------|---------|---------------------|
| 字段映射歧义 | 无法确定两个字段是否为同一含义 | 提前指定 `字段映射` |
| 关联键不确定 | 自动探测的关联键可能不完整或不正确 | 提前指定 `关联键` |
| 数值显著差异 | 同一含义字段在不同数据源中数值差异超过 20% | — |
| 数据结构异常 | 某数据源列数/行数与其他源严重不对等 | — |
| 数据源大面积空值 | 某个数据源的行或列大面积（如 >50%）为空值 | — |
| 参数矛盾 | 用户指定与实际数据明显冲突 | 修正参数 |
| 编码检测失败 | 某文件经多种常见编码尝试后仍无法正确读取（乱码/解码错误） | 由用户提供正确编码或提供重新导出的文件 |
| 时间范围严重不重叠 | 多源 join 时，任意两数据源的时间重叠不足任一方时间跨度的 50% | 确认是否继续或先进行时间范围对齐筛选；用户可通过提前声明"时间范围不需确认"来避免被询问 |

## 4. 输出约定

**用户对输出格式或内容有明确要求时，严格按用户指令执行。未指定部分按以下默认行为：**

- 当 `输出方式` 为默认（单文件）时：输出单个汇总文件（xlsx），包含 **数据表** + **问题清单**（传递给下一环节「数据清洗」）
- 当 `输出方式` 为按维度拆分时：输出多个文件，每个文件包含该维度下的 **数据表** + **问题清单摘要**（仅含该分片内的问题），同时在父目录生成一份 **全局问题清单** 汇总所有分片的问题
- 内部计算使用标准类型（浮点数、日期对象），但写入文件时**单元格显示格式**与输入保持一致（除非用户另行指定）；多输入格式不统一时以第一个文件为基准
- 分类字段的空值变体（空字符串、"-"、"N/A"、"——" 等）统一转为标准空值
