# 数据清洗规则

## 核心原则

**用户指定优先，未指定时 Agent 自主推断。**

本环节接收原始数据文件（可能来自「数据汇总」环节的输出，也可能由用户直接提供待清洗的数据文件），执行数据清洗后输出干净的数据表。仅列出需要 Agent 遵守的约束规则，基础清洗能力由 Agent 自主完成。

---

## 1. 用户可指定的参数

| 参数 | 说明 | 未指定时的默认行为 |
|------|------|-------------------|
| `清洗步骤` | 指定要执行的清洗环节：`缺失值` / `异常值` / `去重` / `类型转换`（可多选，如"只做去重和异常值处理"） | 未指定时进入自动检测模式（§2），数据支持哪些就做哪些 |
| `清洗范围` | 指定对哪些字段/行执行清洗（如"只清洗数值列"、"跳过某字段"） | 对全部字段和行执行清洗 |
| `缺失值处理方式` | 指定如何处理空值：`删除` / `填充`（可指定填充值，如0/均值/中位数/前后值）/ `保留原样` | 未指定时遇缺失值必须逐一询问用户 |
| `异常值判定标准` | 指定什么算异常值（如">3倍标准差"、">99分位数"、或具体阈值） | 结合统计方法自动识别 |
| `异常值处理方式` | 指定异常值的处理方式：`保留并标注` / `修正`（截断/替换）/ `删除` | 保留并标注，在问题清单中记录 |
| `去重规则` | 指定去重的键字段和策略（保留首行/末行/手动选择） | 基于全行内容去重，保留首行 |
| `数据类型转换` | 指定某些字段的类型转换规则（如"将'1.23万'转为数值"） | 与原文件保持一致，不主动转换 |
| `输出格式` | `xlsx` / `csv` / `嵌入报告` / `纯文本摘要` | 与输入文件保持一致；未指定时默认 xlsx |
| `输出内容` | 指定输出内容：`干净数据表` / `清洗报告` / `两者都要` | **两者都要** |
| `单元格格式` | 数值/日期等单元格的显示格式 | 与输入文件保持一致；多输入格式不统一时以第一个文件为基准 |

---

## 2. 清洗执行范围决策

**本节决定执行哪些清洗步骤（§3-§6）。必须在进入任何具体清洗操作之前先完成此决策。**

### 2.1 决策流程

```
用户是否通过 `清洗步骤` 参数指定了要执行的清洗环节？
├── 是 → 仅执行指定的步骤，跳过其他步骤
│   ├── 某步骤的数据前提不满足 → 向用户说明原因并给出替代方案（如"数据无缺失值，跳过缺失值处理"）
│   └── 某步骤的数据前提满足 → 执行
│
└── 否 → 进入自动检测模式（§2.2）
```

### 2.2 自动检测模式（用户未指定 `清洗步骤` 时）

按以下顺序逐项检测数据是否存在该类问题，**检测到则纳入执行计划，未检测到则跳过**：

| 顺序 | 清洗步骤 | 触发条件（任一满足即执行） | 跳过条件 |
|------|---------|-------------------|---------|
| **0** | **脏数据过滤（§2.3）** | **探查中发现测试用户、虚拟号段、明显假数据等脏数据** | **未发现任何脏数据特征** |
| 1 | 缺失值处理（§3） | 数据中存在任何空值（NaN/空字符串/"-"/"N/A"等变体） | 数据完全无空值 |

> **注意**：自动检测仅负责"发现缺失值并纳入执行计划"，具体处理方式仍需按 §3 规则在执行前向用户确认（除非用户已通过 `缺失值处理方式` 参数提前指定）。即：检测是自动的，决策是确认的。
| 2 | 异常值处理（§4） | 存在统计异常（>3σ 或超出 IQR）、业务不合理值（负数曝光等）、或格式异常（类型错误） | 各项检测均未发现异常 |
| 3 | 去重（§5） | 存在完全重复行或疑似重复行 | 无任何重复行 |
| 4 | 类型转换（§6） | 存在可自动修复的类型问题（如数值列存为文本、"1.23万"等带单位符号的数值） | 所有字段类型均正确且无需转换 |

> **输出要求**：无论最终执行了哪些步骤，**必须在清洗报告中列出实际执行的步骤清单和跳过的步骤及原因**，让用户清楚知道做了什么、没做什么、为什么。

### 2.3 脏数据与测试数据过滤【强制前置步骤】

**本步骤在所有其他清洗操作之前执行。不可跳过。**

#### 2.3.1 探查阶段：必须主动扫描的数据特征

在读取数据后，**必须逐行扫描以下特征**，不能仅依赖统计指标判断：

| 扫描维度 | 具体检查项 | 示例 |
|---------|-----------|------|
| **用户ID异常** | ID是否包含测试标识、是否偏离正常编号规律、是否存在明显占位符 | `U99999`、`test_001`、`XXX`、`dummy`、`placeholder`、`TEMP` |
| **姓名/名称异常** | 是否包含测试/假数据关键词 | 含「测试」「test」「demo」「sample」「mock」「假」「虚拟」「临时」等（不区分大小写） |
| **联系方式异常** | 手机号/邮箱是否为虚拟号段或无效格式 | `1390000xxxx`、`1380000xxxx`、连续≥7位相同数字、`10086`变体、`12345678901` |
| **编号序列断裂** | 用户ID是否存在远离主集合的孤立值 | 主集合为U00001~U00050，突然出现U99999 |
| **关键字段全空** | 用户标识字段为空的行 | user_id/user_name 为 NaN 或空字符串 |

#### 2.3.2 判定标准

满足以下**任一条件**即判定为脏数据/测试数据，**直接剔除，无需询问用户**：

| 条件 | 说明 |
|------|------|
| 用户ID或姓名字段包含测试/假数据关键词 | 「测试」「test」「demo」「sample」「mock」「dummy」「假」「虚拟」「XXX」「TEMP」「placeholder」及其常见变体 |
| 联系方式（手机号/邮箱）属于明显的虚拟号段 | 连续相同数字≥7位、`0000`/`1111`/`8888`等测试常用前缀、`123456789`等有序递增模式 |
| 用户ID严重偏离正常编号序列 | 如正常为U00001~U00050，出现U99999这种数量级跳变的孤立值 |
| 关键用户标识字段为空 | 用户ID/姓名等唯一标识字段为空 |

> **判定原则**：宁可误判（把可疑数据标记出来），不可漏判（让测试数据混入分析结果）。当某条记录同时命中多个特征时（如ID=U99999 + 姓名=测试用户B + 手机号=13900002222），**确信度为极高，直接剔除**。

> **优先级说明**：本节（§2.3脏数据过滤）的判定优先于 §4（异常值处理）。即：若某行记录同时被 §2.3 判定为脏数据又被 §4 判定为异常值，**以脏数据规则为准直接剔除**，不再走异常值的"保留并标注"流程。

> **上限保护**：若脏数据剔除量超过原始总行数的 **20%**，则不直接全部剔除，而是暂停并向用户确认（给出被标记的脏数据清单及占比），由用户决定是全量剔除、部分保留还是调整判定标准后重新执行。

#### 2.3.3 处理方式

- **直接剔除**该行数据，不计入后续任何分析和统计
- 在清洗报告中**明确记录**被剔除的行号、用户ID、剔除原因（如"第X行 U99999：用户ID含'99999' + 姓名=测试用户B + 手机号为虚拟号段13900002222"）
- 后续所有统计（人数、占比、均值等）均以**过滤后的有效数据**为准

## 3. 缺失值处理规则

以下情况**必须**按此表处理：

| 场景 | 处理方式 |
|------|---------|
| 存在任何缺失值 | **必须向用户询问**处理方式（删除 / 填充及填充值 / 保留原样），不得自行推断 |
| 用户明确指定了缺失值处理方式 | 严格按用户指定执行，不再逐一询问 |
| 缺失比例超过 30% 的列 | 无论用户选择何种处理方式，均**必须在问题清单中重点标注**，提示用户考虑是否该列本身无意义 |

---

## 4. 异常值处理规则

### 4.1 判定依据

- **统计异常**：偏离均值超过 3 倍标准差、或超出 1.5×IQR 范围
- **业务不合理**：负数的曝光量/点击量、超过 100% 的非比率指标、明显量级错误的数值（如花费 999999）
- **格式异常**：数值列中出现文本、日期列中出现乱码等类型错误

### 4.2 处理要求

| 场景 | 处理方式 |
|------|---------|
| 统计层面的异常值 | **默认保留原值不变**，在问题清单中备注位置和原因；用户要求修正时才执行修正 |
| 业务层面明显错误（如负数曝光） | **必须向用户确认**是数据错误还是特殊业务场景后再处理 |
| 格式异常（类型错误） | 尝试修复（如去除单位符号后转数值）；**无法修复时必须向用户询问**如何处理 |
| 同一维度下连续多行均为异常值 | 可能是该维度的真实特征而非异常，**降低敏感度**，仅在问题清单中备注 |

---

## 5. 去重规则

| 场景 | 处理方式 |
|------|---------|
| 完全重复行（所有字段值相同） | **默认删除**，保留第一条；用户指定了保留规则时按用户指定 |
| 仅部分维度字段重复（如同一日期+渠道出现多次） | **默认保留**，在问题清单中标注；可能代表不同子维度（不同创意/计划），需用户确认是否聚合 |
| 汇总阶段已记录的重复行 | 按本节规则统一处理，不再单独提示 |

---

## 6. 必须确认的场景

以下情况**必须暂停并向用户确认**。确认时**必须同时给出建议的替代方案**供用户选择，而非仅抛出问题：

| 场景 | 触发条件 | 确认时应给出的替代方案 | 用户可如何避免被询问 |
|------|---------|---------------------|---------------------|
| 存在缺失值 | 数据中检测到任何空值 | 列出各缺失字段及缺失率，推荐处理方式（如"列 A 缺失 2% 建议均值填充；列 B 缺失 45% 建议考虑删除该列"） | 提前全局指定 `缺失值处理方式`（则不再逐一询问） |
| 高缺失率列 | 某列缺失率 >30% 或整列为空 | 给出三个选项：① 删除该列 ② 用填充值补全并标注 ③ 保留原样并在报告中重点提示 | 同上 |
| 大面积删除风险 | 按用户指定的规则将删除超过总行数 20% 的数据 | 给出替代方案：① 改用保留策略（如用填充代替删除）② 仅删除极端行（缩小判定范围）③ 分批处理先看效果 | 调整处理方式 |
| 业务不合理值 | 出现违反业务常识的值（负数曝光、CTR>100% 等）且无法判断是否为特殊场景 | 列出具体异常值及其位置，给出选项：① 标记为错误并修正 ② 保留但标注为需人工复核 ③ 确认为特殊业务场景不予处理 | — |
| 去重歧义 | 部分维度重复但不完全相同，无法确定是否为重复数据 | 展示重复行的具体差异字段，推荐方案（如"这些行仅在'创意名称'上不同，建议保留——可能代表不同创意素材"） | 提前指定 `去重规则` 和关联键 |
| 清洗前后差异过大 | 清洗后数据量或统计分布变化幅度超过预期 | 展示清洗前后的关键指标对比（行数、均值、分布），建议用户确认是否继续或回退部分操作 | — |

---

## 7. 输出约定

**用户对输出格式或内容有明确要求时，严格按用户指令执行。未指定部分按以下默认结构输出：**

- 输出一份数据文件（xlsx），包含：**干净数据表** + **清洗报告**
- **清洗报告**内容包括：
  - 实际执行的步骤清单和跳过的步骤及原因
  - 各字段的处理操作摘要（删除了多少空值行、修正了多少异常值、去除了多少重复行、剔除了多少脏数据等）
  - 仍残留的问题清单（传递给下一环节「数据分析」）
  - 数据量变化对比：清洗前 N 行 → 清洗后 M 行（变化率 X%）
- 单元格显示格式与输入保持一致（除非用户另行指定）

> **与主流程输出约束的协调**：当 SKILL.md 的「输出约束」规定"仅交付用户要求的文件，不得附带过程文档"时（如用户明确说"只要csv结果"），本节的清洗报告仍需在内部生成用于质量追溯，但最终交付给用户时可省略。即：**内部过程必须完整记录，外部交付按用户要求精简。**
