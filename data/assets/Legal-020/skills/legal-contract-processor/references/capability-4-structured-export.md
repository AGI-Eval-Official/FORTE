# 能力四：合同数据结构化导出

## 功能说明

将合同中的关键信息提取并导出为结构化数据格式，便于后续系统集成、数据分析或其他程序化处理。

## 输入输出（默认）

> **用户可在 input 中指定具体的输入文件、输出格式、数据结构/Schema、字段范围等。以下为无特殊指定时的默认行为。**

| 类型 | 格式 | 说明 |
|------|------|------|
| 输入 | .docx | 合同文件 |
| 输出 | .xlsx 或 .json | 结构化数据（默认 Excel，用户可指定 JSON） |

### 默认导出结构（示例）

当用户未指定数据结构时，默认按以下框架导出：

**Excel 格式**：单 Sheet 表格，每行一个字段，包含：数据字段名、字段值、所在条款、备注说明

**JSON 格式**：层级化结构，包含以下模块：
- `contract_meta`：合同基本信息（名称、编号、日期、类型、期限）
- `parties`：当事方信息（甲乙方各自的公司名称、地址、联系人、银行账户等）
- `subject_matter`：标的信息
- `financial_terms`：财务条款（总价、支付方式、付款节点、发票、税率）
- `performance_terms`：履行条款（期限、交付、验收）
- `liability_terms`：责任条款（违约金、赔偿上限）
- `dispute_resolution`：争议解决
- `other_terms`：其他条款（保密、知识产权、不可抗力、终止、附件等）

> 若用户在 input 中指定了不同的数据结构、字段列表、输出格式或编码要求，则以用户指定为准。

## 导出规则

1. 提取的字段尽可能完整，空值字段保留键名但值为 null
2. 金额字段保留原始表述（同时注明大写和小写如有）
3. 日期字段统一为 YYYY-MM-DD 格式
4. JSON 输出使用 UTF-8 编码

## 兜底机制

### 场景一：字段配置匹配失败
- 回退到通用商业合同字段配置；仍失败则进入**自由提取模式**，JSON 输出中添加 `_meta.source_mode` 标记

### 场景二：某字段在合同中不存在
- 值设为 `null`，备注填写「合同中未涉及此项」

### 场景三：同一字段出现多份不同值
- 取正文主条款值为主值，备注中列出冲突值并提示人工核对

### 场景四：含特殊字符无法编码
- 必要转义；实在无法编码的内容设为 `null` 并备注

## 执行步骤

1. 解析用户 input → 提取用户指定的输出格式/数据结构/字段范围（若有），否则采用默认方案
2. 读取合同文件（解析失败则报错终止）
3. 加载字段配置作为提取参照（默认使用系统集成配置，失败则触发兜底场景一）
4. 按用户指定或默认的数据结构逐字段提取
   - 字段不存在于合同 → 触发兜底场景二
   - 同一字段多处值冲突 → 触发兜底场景三
5. 对无法精确匹配的字段，尝试通过语义理解提取
6. 准备输出数据（JSON 特殊字符问题则触发兜底场景四）
7. 生成文件并保存
