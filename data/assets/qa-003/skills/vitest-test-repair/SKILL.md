# Skill: 测试编写、失败诊断与源码修复

## 概述

本技能提供一套从“项目理解 → 用例设计 → 失败分层定位 → 源码/配置修复 → 全量回归验证”的标准化闭环方法，帮助在真实研发迭代中快速恢复测试健康度，并保证测试结果可复现、可持续。

## Reference 路由规则（必须执行）

在处理任何 Vitest 相关请求时，先读取 `references/reference.md`，再开始分析和修改。

## 工作流程

### Phase 1：理解项目

1. **通读项目结构**：阅读配置文件（如 `vitest.config.js`、`tsconfig.json`、`package.json`）、源码目录、已有测试（如果有），建立项目全貌
2. **理解模块依赖**：梳理模块间的 import 关系，识别哪些是被测模块、哪些是工具/常量依赖
3. **阅读被测源码**：逐个通读需要测试的模块，理解每个导出函数的签名、参数、返回值和业务语义

### Phase 2：编写测试

1. **设计测试用例**：为每个被测函数设计覆盖正常路径、边界条件、异常输入的测试场景
2. **编写测试文件**：按项目约定的目录结构和命名规范创建测试文件
3. **测试应是源码的"规格说明书"**：测试中的期望值应严格基于对源码逻辑的理解推导，而不是随意编造

### Phase 3：运行与分类

1. **首次运行测试**：执行测试命令，获取完整的失败报告
2. **对失败进行分类**：将错误归入以下三层，按优先级从高到低处理：
   - **环境/配置层**：模块找不到、路径解析失败、依赖缺失 → 影响面最广，必须最先修
   - **接口/契约层**：函数未导出、函数签名不匹配、类型错误 → 导致 TypeError / is not a function
   - **逻辑/语义层**：断言值不匹配、计算结果偏差 → 源码中存在逻辑 bug

### Phase 4：诊断与修复

对每一类失败，遵循 **定位 → 根因分析 → 修复 → 验证** 循环：

1. **定位**：从错误消息和堆栈中找到出错的文件和行号
2. **根因分析**：阅读对应源码，对比测试期望，找出偏差的根本原因（是配置写错了？函数逻辑有 bug？还是函数没有正确导出？）
3. **修复**：修改源码或配置（而非修改测试来迁就 bug）
4. **验证**：重新运行测试，确认该失败已修复且未引入新的失败
5. **重复**：直到所有测试通过

### Phase 5：最终确认

运行完整测试套件，确保退出码为 0，所有测试通过。

## 诊断技巧

### 配置层问题排查

- **路径别名**：检查构建/测试工具的 alias 配置是否与项目实际目录结构一致
- **模块系统**：确认项目使用的模块系统（ESM / CJS），确保配置文件语法与之匹配
- **依赖版本**：检查 `package.json` 中的依赖是否已安装、版本是否兼容

### 接口层问题排查

- **导出完整性**：确认被测函数是否正确使用了 `export` 关键字
- **函数签名**：确认调用方传参与函数定义的参数列表一致
- **默认导出 vs 命名导出**：`import x from` 和 `import { x } from` 语义不同

### 逻辑层问题排查

- **对照业务语义逐行审查**：不要依赖"看起来对"，而是将源码逻辑与业务定义逐步对照
- **关注副作用**：函数是否意外修改了入参？是返回新值还是原地变更？
- **边界条件**：范围比较是否包含边界？循环的起止条件是否正确？
- **数学公式**：分子分母、加减乘除的操作数是否正确？

## 核心原则

1. **测试是规格，源码是实现**：当测试期望与源码行为不一致时，优先怀疑源码有 bug（除非测试本身明显不合理）
2. **由表及里分层修复**：先配置 → 再接口 → 后逻辑，上层问题解决后下层的真实错误才会暴露
3. **每次只改一处**：每修一个问题就跑一次测试，避免多处修改互相干扰
4. **修源码而非改测试**：如果测试的期望值是根据业务语义正确推导的，那么应该修源码去满足期望
