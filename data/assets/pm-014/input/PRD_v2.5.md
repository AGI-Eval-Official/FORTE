## 背景与目标

本次 v2.5 迭代聚焦三个方向：

1. **内容安全**：新增 AI 输出的合规拦截能力，满足监管要求
2. **实验能力**：新增 A/B 测试管理，支持模型版本对照实验
3. **国际化**：新增多语言配置，支持海外市场接入

## 上线条件

| 条件 | 标准 |
|------|------|
| P0 功能提测率 | 100%（所有 P0 功能必须完成提测） |
| P0 功能测试通过率 | 100%（结论为「通过」或「有条件通过」） |
| P1 功能完成率 | ≥ 80%（结论为「通过」的 P1 功能数 / P1 已提测功能总数） |
| 「严重」等级问题 | 清零，不得带病上线 |

以上条件须**同时满足**，方可上线。

## 质量校验规则

> 本节定义上线合规校验中各类问题的判定标准、问题严重程度划分（严重 / 主要 / 次要），以及对应的处置策略。

---

### 一、问题严重程度等级定义

问题严重程度统一分为 **3 级**：严重、主要、次要。

| 等级 | 名称 | 定义 | 典型场景 |
|------|------|------|----------|
| 🔴 严重 | 严重 | 核心功能完全不可用、数据丢失/错误、系统崩溃，或安全漏洞；P0 功能的任何问题均归入此级 | P0功能任意不通过原因；实验数据看板依赖超时无法加载；安全审核接口鉴权失效；非P0功能兼容问题出现在 3 个及以上线上环境 |
| 🟠 主要 | 主要 | 功能降级但有绕过方案，或影响范围较大的体验问题；非P0功能的功能性问题类型 | 非P0功能缺陷、性能问题、数据一致性问题、依赖异常|
| 🟡 次要 | 次要 | 影响范围有限的体验缺陷或边缘场景问题，不影响核心使用；非P0功能的视觉/交互类问题 | 非P0功能界面缺陷、体验缺陷、兼容性问题（仅1个环境出现「部分通过」或「兼容问题」）；超时无提醒；字体渲染异常 |

---

### 二、问题严重程度判定规则

#### 规则一：P0 功能问题一律为「严重」

**无论不通过原因类型是什么，P0 功能存在任何问题（功能缺陷、性能问题、兼容问题等），均直接判定为「严重」。**

#### 规则二：非 P0 功能按原因类型定级

| 不通过原因类型 | 基础等级 | 说明 |
|---------------|---------|------|
| 功能缺陷 | 🟠 主要 | 功能逻辑不符合 PRD 描述，修复后重新提测 |
| 性能问题 | 🟠 主要 | 响应超时、加载失败等影响核心功能可用性 |
| 数据一致性问题 | 🟠 主要 | 数据写入/读取出现错误、负值、丢失等 |
| 依赖异常 | 🟠 主要 | 第三方服务或内部组件不可用导致功能失效 |
| 界面缺陷 | 🟡 次要 | 布局错乱、样式溢出、元素遮挡等视觉问题 |
| 体验缺陷 | 🟡 次要 | 交互不符合预期，但不影响功能完整性 |
| 兼容性问题 | 🟡 次要（基础） | 特定环境下的渲染/交互异常，见兼容升级规则 |
| 其它问题 | 🟡 次要（基础） |除以上不通过原因类型的其它问题|

#### 规则三：兼容性问题的等级升级规则

| 触发条件 | 升级后等级 |
|----------|-----------|
| 同一功能在 **3 个及以上**线上环境出现「部分通过」或「兼容问题」 | 🔴 **升级为严重**，其余均为次要 |

> **说明**：风险等级按照规则一、规则二、规则三的顺序依次判定，同一功能仅纳入一类风险等级。

### 三、完成率计算方法


**功能提测覆盖率**

```
提测覆盖率 = 已提测功能数 / PRD功能总数 × 100%
```

**功能用例通过率**

```
用例通过率 = 通过用例数 / 用例总数 × 100%
```

**功能完成率**

```
功能完成率 = (测试结论为「通过」的功能数) / 已提测功能总数 × 100%
```

> 测试结论为「待测」「不通过」「有条件通过」的功能，不计入分子。

**综合用例通过率**

```
综合通过率 = Σ(各功能通过用例数) / Σ(各功能用例总数) × 100%
```

> 「待测」功能用例数计为 0，不纳入分母；「不通过」功能完整计入分母。
