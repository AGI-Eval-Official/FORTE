# 产品功能说明书生成器

## 适用边界

### 触发场景
**操作类型**（二选一）：
- **创建文档**：编写/生成产品功能说明书、产品说明文档、产品手册、用户使用指南、功能介绍文档
- **更新文档**：更新现有说明书内容，支持三种更新模式
  - PRD→手册更新：根据 MEP 需求链接或 PRD 文本，自动分析需更新章节
  - 功能 Diff 诊断：对比产品现状与手册内容，输出过时/缺失/不一致报告
  - 用户指令更新：按用户明确指令执行更新

**版本管理**（默认开启，除非用户明确说"不要版本管理"）：
- 创建文档时自动初始化版本号为 V1.0
- 更新文档时自动升级版本号并记录修订内容
- 支持单独执行版本管理操作：升级版本号、输出变更记录、保留历史版本

用户表达示例："给这个产品写一份说明书"、"根据这个 PRD 更新手册"、"诊断一下手册是否过时"、"把登录章节改成短信验证码登录"

### 不触发场景
- 明确要求 PRD（产品需求文档）
- 明确要求技术方案、接口设计、研发实现细节文档
- 明确要求营销文案、品牌宣传稿

## 工作流程路由

### Step 0：识别操作类型与版本管理需求

根据用户输入，判断本次操作：

**操作类型**（必选其一）：
- **创建文档**：用户首次编写说明书，或明确说"写一份/生成一份/新建"
- **更新文档**：用户提供已有说明书文件，要求修改、补充、同步变更

**版本管理开关**（默认开启）：
- 开启：用户未明确说"不要版本管理"/"不用管版本"
- 关闭：用户明确说"不要版本管理"/"不用管版本"/"只更新内容不改版本号"

### Step 1：路由到对应规则文件

根据操作类型，读取对应的规则文件并执行：

| 操作类型 | 规则文件 | 说明 |
|----------|----------|------|
| 创建文档 | `reference/create-rules.md` | 输入收集、章节裁剪、内容生成、输出格式 |
| 更新文档 | `reference/update-rules.md` | 三种更新模式、diff 草稿、诊断报告 |
| 版本管理 | `reference/version-rules.md` | 版本号规范、升级判断、修订记录 |

### Step 2：按需加载平台补充规则

根据产品类型，额外读取平台补充规则：

| 产品类型 | 补充规则文件 |
|----------|--------------|
| 网站 / Web 应用 | `reference/web-rules.md` |
| APP（iOS / Android） | `reference/app-rules.md` |
| 微信小程序 | `reference/miniprogram-rules.md` |
| 多端产品 | 读取所有相关平台文件 |

### Step 3：执行并输出

按规则文件执行完整流程，输出最终文档。

```
用户请求
    ↓
识别操作类型（创建/更新）
    ↓
识别版本管理开关（默认开启）
    ↓
读取 product-manual-rules.md（通用基础规范）
    ↓
┌─────────────────────────────────────┐
│ 操作类型 = 创建？                     │
│   ├─ 是 → 读取 create-rules.md       │
│   │         ↓                        │
│   │       收集输入 → 裁剪章节 →       │
│   │       生成内容                   │
│   │         ↓                        │
│   │       版本管理开启？              │
│   │         ├─ 是 → 初始化 V1.0.0    │
│   │         └─ 否 → 使用日期命名     │
│   │         ↓                        │
│   │       输出文档                   │
│   │                                  │
│   └─ 否 → 读取 update-rules.md       │
│             ↓                        │
│           识别更新模式（PRD/诊断/指令）│
│             ↓                        │

│             ↓                        │
│           版本管理开启？              │

│             ↓                        │
│           输出文档                   │
└─────────────────────────────────────┘
```

## 写作原则

1. 以用户视角描述"怎么用"，不写研发实现细节
2. 用词统一、结构清晰、可快速扫读
3. 模板可裁剪，不机械套章
4. 结论和规则可验证，避免空泛描述

## 执行约束

### 输出文件约束

用户对输出文件有明确要求时，**严格遵守**：

| 用户要求类型 | 遵守规则 |
|--------------|----------|
| 输出格式 | 必须使用指定格式（Word/Markdown/HTML/PDF），不得擅自更改 |
| 输出路径 | 必须输出到指定路径，不得更改位置 |
| 文件命名 | 必须使用指定文件名，不得添加后缀或版本号（除非用户要求版本管理） |
| 输出数量 | 仅输出用户要求的文件，不得额外生成预览、草稿、备份文件 |

**示例**：
- 用户要求「输出到桌面」→ 必须输出到桌面路径
- 用户要求「文件名叫产品说明」→ 文件名必须为 `产品说明.docx`，而非 `产品说明_草稿.docx`
- 用户要求「只输出 Markdown」→ 不得同时生成 Word 或其他格式

### 文件清理约束

任务结束时，**必须清理**以下文件：

| 清理类型 | 说明 |
|----------|------|
| 中间文件 | diff 草稿、诊断报告、临时合并文件等非最终交付物 |
| 预览文件 | 用于用户确认的临时预览文档 |
| 备份文件 | 自动生成的备份副本（如 `xxx_backup.docx`） |
| 草稿文件 | 生成过程中的中间草稿 |

**保留文件**：
- 用户明确要求保留的文件
- 最终交付的输出文档
- 用户原有的输入文件（不得删除或覆盖，除非用户明确要求）

### 确认机制

以下场景需用户确认后再执行：

| 场景 | 确认内容 |
|------|----------|
| 覆盖现有文件 | 「文件已存在，是否覆盖？」 |
| 删除文件 | 「是否删除中间文件 xxx？」 |
| 输出路径不存在 | 「路径不存在，是否创建？」 |
