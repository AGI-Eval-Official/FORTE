# 新建文档规则

本文档定义产品功能说明书的创建流程、输入要求、内容生成规范和输出格式。

> **前置依赖**：执行本文档规则前，需先读取 `product_manual_rules.md` 获取通用章节结构、编写规范和自检清单。

---

## 输入资料清单

在开始创建文档前，优先使用用户已提供资料；缺失项再补充收集。

### 必填信息

| 信息项 | 必填 | 说明 |
|--------|------|------|
| 产品名称 | ✅ | 文档封面、页眉、文件名使用 |
| 产品类型 | ✅ | 网站 / APP / 小程序 / 后台系统 / 桌面客户端 / 硬件配套 / 其他 |
| 目标读者 | ✅ | 客户 / 终端用户 / 合作伙伴 / 内部团队 |
| 功能清单 | ✅ | 模块与功能点列表，支持先粗后细 |

### 常见参考资料

用户可能提供的参考资料，按优先级排序：

| 优先级 | 资料类型 | 内容说明 | 用途 |
|--------|----------|----------|------|
| ⭐⭐⭐ | 产品 Demo / 测试环境链接 | 可操作的产品环境 | 验证功能、录制操作、截取界面 |
| ⭐⭐⭐ | 原型图 / 交互稿 | Figma/Axure/Sketch 链接 | 界面布局、操作流程、交互细节 |
| ⭐⭐⭐ | PRD / 需求文档 | MEP 链接或文档文件 | 功能逻辑、边界条件、特殊场景 |
| ⭐⭐ | 界面截图 / 操作录屏 | 功能界面截图或操作演示视频 | 补充细节、确保步骤准确 |
| ⭐⭐ | 旧版说明书 | 已有文档文件 | 保持风格一致、参考结构 |
| ⭐ | 用户反馈 / FAQ | 用户高频问题汇总 | 补充常见问题章节 |
| ⭐ | 竞品说明书 | 同类产品文档 | 参考结构、表述方式 |

### 资料收集话术

当用户未提供完整资料时，按以下方式收集：

**基础信息**：
- "请问这个产品的名称是什么？"
- "请问这是什么类型的产品？（网站 / APP / 小程序 / 后台系统 / 桌面客户端 / 硬件配套）"
- "这份说明书是给谁看的？（客户 / 终端用户 / 合作伙伴 / 内部团队）"

**功能资料**：
- "请列出这个产品的主要功能模块和功能点。"
- "有产品 Demo 或测试环境链接吗？我可以直接操作验证。"
- "有原型图或 PRD 文档吗？方便我理解功能细节。"

**补充资料**：
- "有用户反馈或常见问题汇总吗？可以补充到 FAQ 章节。"
- "有旧版说明书吗？可以保持风格一致。"

## 文档类型判断

### 判断规则

在开始写作前，先判断用户要的是"功能说明书"而不是其他文档类型：

| 文档类型 | 特征 | 处理方式 |
|----------|------|----------|
| **功能说明书** ✅ | 描述产品如何使用、操作指南 | 按本规则执行 |
| PRD（产品需求文档）❌ | 描述产品应该做什么、需求规格 | 不触发本 skill，提示用户 |
| 技术方案/接口设计 ❌ | 描述技术实现细节 | 不触发本 skill，提示用户 |
| 营销文案 ❌ | 宣传推广内容 | 不触发本 skill，提示用户 |

### 歧义处理

若存在歧义，优先按"功能说明书"执行，并在文稿中：
- 避免研发实现细节
- 使用用户视角描述"怎么用"
- 不涉及技术架构和接口设计

## 输出格式

根据用户需求和使用场景，支持多种输出格式：

### 输出格式选择

| 格式 | 适用场景 | 说明 |
|------|----------|------|
| **Word (.docx)** | 正式交付、打印、存档 | 默认格式，完整排版 |
| **Markdown (.md)** | 内部协作、版本管理、在线发布 | 轻量级，易于编辑 |
| **HTML** | 在线帮助中心、网站发布 | 可直接部署到 Web |
| **PDF** | 正式交付、不可编辑分发 | 导出后不可修改 |
| **纯文本** | 快速预览、邮件发送 | 无格式，仅内容 |

用户未指定时，默认输出 Word (.docx) 格式。

---

### Word 文档格式

使用 docx-js 输出，遵循以下规范：

**页面设置**：
- 纸张：A4
- 边距：上下左右各 2.54cm

**页眉**：
- 格式：`[产品名称] V[版本号]` 或 `[产品名称]`（版本管理关闭时）
- 字体：宋体 9pt
- 位置：右对齐

**页脚**：
- 内容：页码
- 格式：第 X 页 / 共 Y 页
- 位置：居中

**目录**：
- 位置：文档开头，修订记录之后
- 格式：自动生成 TOC
- 深度：三级标题

**字体规范**：
| 元素 | 中文字体 | 英文字体 | 字号 | 样式 |
|------|----------|----------|------|------|
| 一级标题 | 宋体/微软雅黑 | Arial | 16pt | 加粗 |
| 二级标题 | 宋体/微软雅黑 | Arial | 14pt | 加粗 |
| 三级标题 | 宋体/微软雅黑 | Arial | 12pt | 加粗 |
| 正文 | 宋体/微软雅黑 | Arial | 10.5pt | 常规 |
| 表格内容 | 宋体/微软雅黑 | Arial | 9pt | 常规 |

**段落格式**：
- 行间距：1.5 倍
- 段前间距：0.5 行
- 段后间距：0.5 行

**表格格式**：
- 边框：单线框，灰色
- 表头：加粗，浅灰背景
- 列宽：按内容自动调整

```markdown
### 1.1 产品简介

```

**格式要点**：
- 使用标准 GitHub Flavored Markdown
- 图片使用相对路径或占位符
- 表格使用管道语法
- 代码块标注语言类型

---

### HTML 格式

适用于在线帮助中心和网站发布：

**输出要求**：
- 生成完整的 HTML 文件，包含 CSS 样式
- 支持响应式布局
- 目录可点击跳转
- 支持搜索功能（可选）

**文件结构**：
```
产品说明书_HTML/
├── index.html          # 主文件
├── assets/
│   ├── style.css       # 样式文件
│   └── images/         # 图片资源
```

---

### PDF 格式

适用于正式交付和不可编辑分发：

**输出方式**：
- 先生成 Word 文档，再转换为 PDF
- 或直接从 Markdown 转换

**格式要求**：
- 保持与 Word 版本一致的排版
- 确保中文字体正确嵌入
- 图片清晰度不降低

### 文件命名

> **引用**：完整命名规则详见 `version_rules.md` 文件命名规则章节。

按版本管理状态命名：
- **版本管理开启**：`[产品名称]_功能说明书_V1.0.0.[扩展名]`
- **版本管理关闭**：`[产品名称]_功能说明书_YYYYMMDD.[扩展名]`

### 输出路径

- 优先使用用户指定路径
- 未指定则输出到当前工作目录

## 快速创建模式

当用户需要快速生成文档框架时，支持精简版输出：

### 触发条件

用户明确要求：
- "生成精简版"
- "只要框架"
- "快速出一份"

### 精简版内容

- 保留完整章节结构
- 功能说明仅包含：功能简介 + 功能入口
- 操作步骤使用占位符
- 不包含常见问题

### 精简版标注

在文档开头标注：

```
【精简版文档】
本文档为精简版框架，操作步骤待补充完整。
```
