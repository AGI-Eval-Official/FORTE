# 漏斗转化分析方法论

本文件定义漏斗转化分析的完整方法论框架，供 pm_metric_rca SKILL 在"漏斗转化分析子路径"中读取使用。

---

## 一、漏斗分析的核心概念

### 什么是漏斗分析

漏斗分析是追踪用户完成某一目标路径的逐步转化过程，通过记录各节点的用户数和转化率，定位流失最严重的环节，为产品优化提供数据依据。

### 核心计算公式

```
逐步转化率 = 下一节点用户数 / 当前节点用户数 × 100%
整体转化率 = 最终节点用户数 / 入口节点用户数 × 100%
流失率 = 1 - 转化率
流失用户绝对数 = 当前节点用户数 - 下一节点用户数
```

### 关键判断标准

最大流失节点的判定**不是**简单地看"转化率绝对值最低"，而是综合考虑：

1. **转化率低于行业参考值的幅度**：偏离越大，优化潜力越大
2. **流失用户绝对数**：流失人数最多的节点往往带来最大业务损失
3. **节点所处位置**：越靠后的节点（如支付），用户意向越强，流失越可惜
4. **优化成本与收益比**：某些节点转化率低但优化成本极高，需权衡

## 二、三步分析框架

### 第一步：整体漏斗概览

**目的**：建立全局视角，识别最大流失节点。

**执行内容**：

1. 列出所有漏斗节点，计算每个节点的绝对用户数
2. 计算相邻节点间的逐步转化率
3. 读取 `reference/funnel_benchmark.md` 获取行业参考转化率
4. 对比实际转化率与参考值，标注每个节点的状态（正常 / 偏低 / 严重偏低）
5. 综合判断，确定"最大流失节点"

**输出格式**：

```
| 节点 | 用户数 | 逐步转化率 | 行业参考值 | 偏离幅度 | 状态 |
|------|--------|------------|------------|----------|------|
| 节点A | 100,000 | 100% | - | - | 起点 |
| 节点B | 35,000 | 35% | 15%-35% | 0% | 正常 |
| 节点C | 8,400 | 24% | 40%-65% | -38% | 严重偏低 |
```

**最大流失节点判定示例**：

> 漏斗路径：曝光 → 点击 → 加购 → 下单 → 支付
>
> 各节点转化率：
> - 曝光→点击：30%（参考 15%-35%，正常）
> - 点击→加购：15%（参考 8%-20%，正常）
> - 加购→下单：24%（参考 40%-65%，严重偏低，偏离 -38%）
> - 下单→支付：85%（参考 70%-90%，正常）
>
> **最大流失节点：加购 → 下单**

---

### 第二步：流失节点深挖

**目的**：针对最大流失节点，从多维度分析流失原因，定位根本问题。

#### 维度 1：用户分群对比

将流失节点按不同用户群体拆分，观察是否存在"结构性问题"（某类用户转化率远低于其他群体）。

**常用分群维度**：

| 分群维度 | 分析目的 |
|----------|----------|
| 新老用户 | 判断是新用户引导问题还是老用户转化问题 |
| 渠道来源 | 判断是否是低质量流量导致 |
| 设备类型 | 判断是否是特定设备的体验问题 |
| 地域分布 | 判断是否与网络环境、区域运营策略有关 |
| 用户价值分层 | 判断高价值用户是否也流失严重 |

**分析方法**：

计算每个子群的转化率，与整体转化率对比。

```
子群转化率 = 该子群到达下一节点人数 / 该子群当前节点人数 × 100%
子群对整体的影响 = (子群转化率 - 整体转化率) × 子群占比
```

**典型发现模式**：

- **所有子群都低**：全局性产品/流程问题
- **某个子群特别低**：针对该子群的定向问题（如新用户引导缺失、某渠道质量差）
- **子群间差异大但整体不低**：正常现象，无需过度优化

---

#### 维度 2：时间分布分析

观察流失是否集中在特定时段，帮助判断"环境类问题"。

**常见时间维度**：

| 时间维度 | 可能发现的问题 |
|----------|---------------|
| 小时分布 | 夜间流失高 → 可能是夜间客服不在线、支付通道维护 |
| 星期分布 | 周末流失高 → 可能是工作日/周末用户需求场景不同 |
| 版本上线时间点 | 版本上线后流失骤增 → 新版本引入问题 |
| 活动/促销期间 | 活动期间流失变化 → 活动设计是否影响了正常转化 |

**分析方法**：

绘制各时段的转化率曲线，观察是否存在"断崖式下跌"或"持续性劣化"。

```
时段转化率 = 该时段内到达下一节点人数 / 该时段内当前节点人数
```

---

#### 维度 3：行为路径分析

观察流失用户在节点前后的典型行为序列，理解"用户在流失前经历了什么"。

**核心问题**：

- 流失用户在该节点停留了多久？（停留时间过短 = 没看懂就离开；过长 = 操作卡顿或犹豫）
- 流失用户的上一次行为是什么？（反复返回 = 犹豫；直接退出 = 流失）
- 流失用户离开后的去向？（去了其他页面 = 导航问题；直接退出 App = 体验问题）

**分析方法**：

- 查看流失用户的 session 路径序列
- 对比"成功转化用户"与"流失用户"的路径差异
- 关注"异常行为信号"：反复点击、页面返回、长时间无操作等

---

#### 维度 4：页面/功能状态排查

检查流失节点对应的页面或功能是否存在已知的技术问题或体验劣化。

**排查清单**：

| 检查项 | 排查方式 | 常见表现 |
|--------|----------|----------|
| 页面加载性能 | 查看 LCP/FCP 指标 | 加载慢 → 用户耐心耗尽直接退出 |
| 页面 Bug / 错误 | 查看错误日志、崩溃率 | 页面白屏 / 按钮点击无反应 |
| 接口成功率 | 查看后端接口监控 | 表单提交失败、数据加载不出 |
| 文案/引导清晰度 | 用户访谈、热力图 | 用户找不到下一步按钮、看不懂操作提示 |
| 第三方依赖 | 检查支付 SDK、地图 SDK 等 | 第三方服务故障导致流程中断 |

---

### 第三步：优化假设提出

**目的**：基于前两层分析，提出可验证的优化方向。

#### 假设结构

每个优化假设应包含三部分：

1. **假设内容**：用户为什么在此节点流失（基于数据的因果推断）
2. **验证方式**：如何用数据或用户研究验证假设是否成立
3. **优化方向**：具体的产品/设计/运营改进措施

#### 假设生成原则

- **优先验证"数据支持最强的假设"**：如果用户分群分析显示"新用户转化率仅为老用户的 1/5"，则优先验证"新用户引导不足"假设
- **优先验证"改动成本最低的假设"**：如果修改文案的成本远低于重构流程，两者假设强度相当时优先验证前者
- **优先验证"影响面最大的假设"**：如果某问题影响 80% 的用户，优先于影响 10% 用户的问题

#### 典型假设示例

**场景**：电商产品"加购 → 下单"转化率从 55% 跌至 24%

| 假设编号 | 假设内容 | 验证方式 | 优化方向 |
|----------|----------|----------|----------|
| H1 | 结算页新增的"凑单提示"干扰了用户决策，导致犹豫后放弃 | A/B 测试：去掉凑单提示 vs 保留，对比转化率 | 简化结算页信息，将凑单提示改为非弹窗弱提示 |
| H2 | 某版本升级后，低端机型结算页加载时长从 1.5s 升至 4s，导致跳出 | 分机型查看转化率，对比加载时长与转化率相关性 | 优化结算页代码性能，对低端机型做降级处理 |
| H3 | 满减门槛从 99 元提高至 149 元，加购商品均价 110 元，用户凑单意愿低 | 查看购物车均价分布，计算距离新门槛的平均差距 | 调整满减门槛梯度，或增加"还差 XX 元享优惠"提示 |

## 三、常见陷阱与规避

### 陷阱 1：忽视"输入质量"，只看"流程问题"

漏斗底部的转化率低，可能不是因为流程有问题，而是因为进入漏斗的用户质量本身就很低。

**规避方法**：分析前先检查流量来源结构是否发生变化，必要时做"相同来源用户的漏斗对比"。

### 陷阱 2：过度优化"转化率绝对值最低"的节点

某节点转化率只有 5%，但如果行业参考值也是 5%，且优化成本极高，则不应作为优先项。

**规避方法**：始终对比行业参考值，计算"偏离参考值的幅度"作为优化优先级依据。

### 陷阱 3：混淆"相关性"与"因果性"

发现"流失用户的平均停留时间是转化用户的 2 倍"，不能直接推出"停留时间长导致流失"——可能是用户在努力寻找某个功能但找不到。

**规避方法**：结合用户访谈、session replay 等定性方法验证数据发现的因果关系。

### 陷阱 4：忽略"幸存者偏差"

只分析"流失用户"的行为，忽略了"成功转化用户"的对比。

**规避方法**：所有分析都应有对照组——对比流失用户与成功用户的差异，而非孤立分析单一群体。

## 四、与其他分析方法的协作

### 与指标拆解树的协作

漏斗分析通常作为"转化类指标"维度拆解的一部分。当目标指标是"整体转化率"时：

1. 先用指标拆解树定位"哪条路径的转化率"出了问题
2. 再对该路径做漏斗分析，定位具体流失节点
3. 最后对流失节点做根因假设

### 与 Cohort 分析的协作

当漏斗分析发现"新用户转化率远低于老用户"时：

1. 用 Cohort 分析验证"新用户转化率下降"是持续趋势还是短期波动
2. 若发现特定注册批次的新用户转化率特别低，追溯该批次的获客渠道和首次体验路径

### 与 A/B 测试的协作

漏斗分析提出的优化假设，最终需要通过 A/B 测试验证：

1. 漏斗分析定位问题节点 → 提出优化假设
2. 设计 A/B 实验 → 在问题节点实施改动
3. 实验结果验证假设 → 确认优化方向有效

---

## 五、输出标准

漏斗分析子路径的输出应包含以下模块，缺一不可：

1. **漏斗概览**：完整节点列表、转化率、行业参考对比、最大流失节点标注
2. **流失节点分析**：用户分群、时间分布、行为路径、页面状态四维度分析
3. **优化假设**：2-3 个假设，每个含假设内容、验证方式、优化方向
4. **数据缺失声明**：若有节点数据缺失或某维度无法分析，需明确标注
