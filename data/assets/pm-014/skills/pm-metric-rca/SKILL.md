# SKILL 目的

对产品经理工作中遇到的业务指标波动进行系统性归因分析，帮助快速定位根因，输出可落地的行动建议。

---

# 路径判断

**在读取任何输入参数之前，先判断输入类型：**

- 提供了 `target_metric` → 进入指标归因分析流程
- 提供了 `funnel-steps` → 直接调用漏斗分析（读取 `reference/funnel-methodology.md`）
- 两者都有 → 先走指标归因流程，维度拆解阶段自动调用漏斗分析
- 两者都没有 → 询问："请告诉我分析的是一个指标（如 DAU、转化率），还是一条用户路径（如注册→激活→付费）？"

---

# 输入要求

| 参数 | 说明 | 适用场景 | 是否必填 |
|------|------|----------|----------|
| `target_metric` | 分析的指标名称，如 DAU、转化率、GMV、留存率 | 指标归因分析 | 二选一必填 |
| `metric_direction` | 波动方向：上升 / 下降 / 持续低位 | 指标归因分析 | 是 |
| `funnel-steps` | 漏斗节点列表，如"曝光→点击→加购→下单→支付" | 漏斗分析 | 二选一必填 |
| `time_range` | 分析时间范围，如"2024-11-01 至 2024-11-07" | 两者皆用 | 是 |
| `baseline` | 对比基准：环比上周 / 同比去年同期 / 版本上线前 N 天 | 指标归因分析 | 是 |
| `user_segment` | 用户分群条件，如"近 30 天新用户"、"全量用户" | 漏斗分析 | 是 |
| `magnitude` | 波动幅度，如"-15%"、"下降约 2 万" | 指标归因分析 | 建议提供 |
| `context` | 同期发生的事件，如版本发布、活动上线、竞品动作 | 指标归因分析 | 建议提供 |
| `data_source` | 数据来源，如数仓表名、BI 看板链接 | 两者皆用 | 建议提供 |

**信息补全话术**：若用户未提供必填参数，回复："我需要确认几个信息才能开始分析：[列出缺失项]，请补充后我们继续。"

# 核心规则

## 一、指标归因三层框架

严格按照以下三层框架执行，不得跳步：

### 第一层：数据真实性核查

在进行业务归因前，必须先排除数据问题导致的"虚假波动"：

- **埋点采集**：埋点是否正常上报，是否有延迟或缺失
- **统计口径**：去重逻辑、时区设置、归因窗口是否发生变化
- **数据管道**：ETL 任务是否正常跑完，是否有数据回刷
- **口径对齐**：用户使用的数据来源与分析口径是否一致

若数据真实性存疑，输出"数据质量疑问"结论，建议用户先核查数据链路，暂停业务归因。

### 第二层：维度拆解定位

数据真实性确认后，按以下维度逐一拆解，找到波动集中的子维度：

- **时间维度**：波动是持续性还是单日突发？是否与特定事件时间吻合？
- **渠道/来源维度**：哪个流量来源贡献了主要波动？
- **用户分层维度**：新用户 vs 老用户，哪层用户波动更显著？（Cohort 视角）
- **地域/设备维度**：是否集中在特定地区或机型？
- **功能/页面维度**：是全局性波动还是某个具体路径的问题？

读取 `reference/metric-dimension-map.md` 获取各指标对应的标准拆解维度与常见根因。如遇不熟悉的术语（如 Cohort、归因窗口、贡献度、Aha Moment 等），读取 `reference/glossary.md` 查阅定义与用法。

**若目标指标为转化类指标**（转化率、支付率、激活率等），在维度拆解阶段读取 `reference/funnel-methodology.md`，将漏斗分析结论作为维度拆解的一部分输出。

### 第三层：根因假设与验证

基于维度拆解结果，提出 2-3 个根因假设，并说明每个假设的验证方式：

- **假设来源**：结合 `context`（同期事件）与维度拆解结论
- **验证方式**：明确说明需要查哪张表、看哪个指标、做什么对比
- **置信度评估**：高 / 中 / 低，并说明判断依据

## 二、兜底规则

- 若用户提供的数据不足以支撑某一层分析，在该层输出"数据不足，建议补充 [具体数据]"，继续执行后续层级
- 若分析后仍无法定位根因，输出"当前信息下无法确定根因，建议排查方向：[列出 2-3 个方向]"
- 若漏斗节点数据不完整（某节点无数据），在该节点标注"数据缺失"，继续分析其他节点
- 若用户未提供分群条件，默认使用"全量用户"并在输出中注明
- 禁止在数据不足时凭空捏造归因结论

# 工作流

## 指标归因分析

```
Step 0：路径判断
  → 确认用户输入属于主路径（有 target_metric）

Step 1：信息确认
  → 检查 target_metric、metric_direction、time_range、baseline 是否齐全
  → 缺失则触发补充询问，等待用户回复后继续

Step 2：数据真实性核查
  → 逐项检查埋点、口径、管道、归因窗口
  → 若存疑 → 输出疑问点，建议用户核查，流程暂停
  → 若无疑问 → 进入 Step 3

Step 3：维度拆解
  → 读取 reference/metric-dimension-map.md
  → 按标准维度逐一分析，标注"波动集中维度"
  → 若目标指标为转化类 → 读取 reference/funnel-methodology.md，将漏斗分析结论并入本步骤输出
  → 输出"维度拆解"模块

Step 4：根因假设与验证建议
  → 基于 Step 3 结论提出 2-3 个假设
  → 每个假设附验证方式与置信度
  → 输出"根因假设"模块

Step 5：行动建议
  → 针对置信度最高的根因，输出 1-2 条可立即执行的行动建议
  → 输出"行动建议"模块

Step 6：Checklist 自检
```

# Checklist（输出前自检）

**指标归因分析**

- [ ] 是否完成了三层归因框架（数据核查 → 维度拆解 → 根因假设），无跳步
- [ ] 根因假设是否有对应的验证方式，而非主观断言
- [ ] 行动建议是否具体可执行，而非泛泛而谈
- [ ] 转化类指标是否已读取 reference/funnel-methodology.md
- [ ] 是否避免了在数据不足时捏造结论

若任一项未通过，重新修正对应模块后再输出。
