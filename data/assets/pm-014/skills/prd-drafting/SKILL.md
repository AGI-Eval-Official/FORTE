# SKILL 目的

通过解析用户的初始输入，结合行业经验进行专业推演，为用户生成结构化、高完成度的产品需求文档（PRD初稿）

核心原则：本 SKILL 只负责需求分析与梳理，不涉及流程图、线框图、状态图等图示绘制。

---

# 路径判断

**在读取任何输入参数之前，先判断用户需求成熟度：**

| 用户输入特征 | 进入路径 |
|-------------|---------|
| 用户能描述清楚功能目标、目标用户等 | 完整路径：直接进入需求梳理 → PRD 生成 |
| 用户能说出大致想法，但细节模糊 | 标准路径：先澄清再梳理 → PRD 生成 |
| 用户只有一句话想法，或明确说"我也没想好" | 兜底路径：输出需求梳理文档（非完整 PRD），帮用户理清思路 |
| 用户上传了已有 PRD，要求补全/优化 | 优化路径：对照 PRD 模板查漏补缺 → 输出补全建议或修订版 |

**兜底路径触发条件**（满足两个以上进入兜底）：
- 用户无法回答"这个功能要解决什么问题"
- 核心流程无法描述清楚（连主流程都没有）
- 目标用户不明确
- 用户明确说"我也没想好"

# 输入要求

执行前需确认以下信息，缺少时主动向用户补充询问：

| 参数 | 说明 | 是否必填 |
|------|------|----------|
| `product_context` | 产品背景或核心问题描述，如"这是一个面向 B 端的审批系统" | 必填 |
| `feature_scope` | 本次 PRD 覆盖的功能范围，如"只写登录和权限管理部分" | 必填 |
| `target_users` | 目标用户/角色，如"运营人员、审核人员、管理员" | 必填 |
| `target_audience` | 文档的主要阅读对象，如"前后端开发"、"测试同学"、"产品评审" | 建议提供 |
| `existing_materials` | 已有的参考材料，如竞品截图、旧版文档、会议纪要 | 建议提供 |
| `constraints` | 硬性约束，如时间要求、技术栈限制、合规要求、对接系统 | 建议提供 |

**信息补全话术**：若用户只说"帮我写个 PRD"而未提供任何背景，回复："好的，在开始之前我需要了解一些基本信息：这个 PRD 是关于什么产品/功能的？本次主要覆盖哪些功能范围？目标用户是谁？"

**如果用户说"你帮我想"或"先按你的理解写"**：可以，但必须做两件事——基于理解给出假设并明确标注为 **[假设]**，在最终输出的"待确认项"中列出所有假设提醒用户逐条确认。

# 核心规则

## 一、PRD 文档结构规范

严格按照五大章节（概述 → 产品描述 → 功能需求详述 → 非功能需求 → 附录）组织 PRD。完整的章节目录树及各章节撰写规范见 `reference/prd-writing-guide.md`。

## 二、用户故事撰写规范

读取 `reference/example-us.md` 对齐用户故事的填写深度。核心要求：

**价值陈述**：必须使用标准格式"作为 [角色]，我希望 [操作]，以便 [目的]"，三要素缺一不可。

**前置条件**：不能只写"用户已登录"，还需说明系统侧的准备状态（如"系统已加载可用的 XX 列表"）。

**Happy Path**：按操作步骤逐条描述，每步说明用户动作和系统反馈，不能只写"用户填写表单并提交"。

**异常处理**：覆盖该故事特有的业务异常。通用异常（网络断开、权限不足、会话过期、并发冲突）写在 2.6 全局说明中，不在每个故事里重复。每种异常需说明触发条件、系统行为、用户可见的提示文案，以及是否保留用户已输入的数据。

**验收标准**：使用 GIVEN-WHEN-THEN 格式，至少覆盖一个成功场景和一个失败/异常场景。THEN 中的期望结果需具体到用户可见的界面变化或文案。

**数据字典**：使用表格格式，包含字段名、类型、是否必填、说明、示例值，缺一不可。

## 三、兜底规则

- 若用户对某个故事的细节无法确定，在该故事中标注"【待确认】[具体问题]"，继续推进其他故事
- 若用户提供的信息存在逻辑冲突，必须指出冲突点并请用户裁决，不得自行选择
- 若用户要求跳过某个确认步骤直接生成，可以执行，但在生成的文档中对跳过确认的部分标注"【需复核】"
- 若触发兜底路径（需求过于模糊），不输出完整 PRD，改为输出"需求梳理文档"（见工作流兜底路径）

# 工作流

## 标准路径

```
Step 0：判断对话模式
  → 默认采用【单轮生成模式】
  → 仅当用户输入极度匮乏（如仅有一句“帮我写个电商PRD”，无任何具体功能点或背景）时，才触发简短提问引导补充；

Step 1：信息解析与提取
  → 从用户描述中提取已知信息：product_context、feature_scope、target_users、大致功能清单。
  → 识别信息缺口（例如：背景、用户、范围、优先级、流程、约束）。
  → 将缺口记录在案，准备在后续步骤中利用产品经理专业经验进行推演和补全，但是需要明确哪些内容是进行假设或者推演的

Step 2：构建业务框架
  → 基于提取的信息，梳理核心业务流程，划分合理的逻辑阶段
  → 若用户提供的流程存在断层，基于行业通用最佳实践（Best Practices）自动补全上下游环节

Step 3：用户故事展开梳理
  → 按阶段顺序，将功能清单逐一转化为标准的用户故事
  → 梳理关键字段业务定义、状态枚举、前置条件、后置结果。
  → 对于用户未提及、完全由 AI 凭借经验推演的关键业务规则、核心字段或流程，在文本旁标注 [需复核]。

Step 4：异常与边界自动补漏
  → PRD 主体梳理完后，对照补漏清单逐项检查：
    - 每个输入字段是否定义了校验规则
    - 每个操作是否定义了失败时的处理
    - 并发操作、数据为空、权限不足、网络异常如何处理
    - 关键页面和操作是否有埋点定义
    - 非功能需求是否至少各有一条
  → 缺少的内容直接补进对应位置，无法判断的标注 [待确认]

Step 5：终稿确认与最终生成
  → 依次读取模板与规范（assets/prd-template.md, reference/prd-writing-guide.md, reference/example-us.md），一次性输出排版精良、结构清晰的完整 PRD 文档。
  → 【尾部附加】在 PRD 正文的最末尾，单开一个「待确认与讨论清单」章节：
    1. 汇总 AI 自动推演的关键逻辑（提醒用户重点 Review 带有 [需复核] 的部分）。
    2. 列出遗留的业务缺口及 AI 的假设前提（如：“关于支付方式您未提及，本次暂按微信/支付宝常规直连模式撰写”）。

Step 6：Checklist 自检
```

## 兜底路径（需求过于模糊时）

```
当触发兜底条件时，不输出完整 PRD，改为输出需求梳理文档：

## [功能名] 需求梳理

### 当前理解
对需求的当前理解，包含所有已知信息

### 待回答的关键问题
按优先级列出需要回答的问题，标注 [必须] / [建议] / [可选]

### 可能的方案方向
列出 2-3 个可能的方案方向，各自优劣

### 建议下一步
具体建议下一步怎么推进

→ 等用户把关键问题答完后，再触发完整的 PRD 生成流程
```

# 输出要求

读取 `assets/prd-template.md` 作为输出的基础框架，严格按模板填充，不得随意调整模块顺序与内容边界。

最终交付两份产出：

**产出一：完整 PRD 文档**，包含五大章节全部内容，所有 [待确认] 和 [假设] 标记保留。

**产出二：待确认项清单**，从 PRD 中提取所有 [待确认] 和 [假设]，按优先级分三档汇总：

```
### 必须确认（阻塞开发）
1. [假设] xxx → 见 x.x 节
2. [待确认] xxx → 见 x.x 节

### 建议确认（影响完整度）
3. [待确认] xxx → 见 x.x 节

### 可后续补充
4. [待确认] xxx → 见 x.x 节
```

**文件命名格式**：`[产品名]_PRD_V[版本号].md`

# Checklist（输出前自检）

| # | 检查项 | 标准 |
|---|--------|------|
| 1 | 背景与目标 | 有业务目标和用户目标，且可量化或可验证 |
| 2 | 角色与权限 | 所有角色已列出，权限边界清晰 |
| 3 | Why（为什么做） | 1.1 背景介绍、1.4 产品目标中有明确的业务问题和目标 |
| 4 | Who（谁来用） | 1.3 目标用户、1.5 角色与权限中角色和权限边界清晰 |
| 5 | What（做什么） | 2.1 需求描述、2.5 功能清单中功能范围明确，有"非目标"说明 |
| 6 | Where（在哪做） | 2.4 产品框架、3.x 功能模块中入口路径和页面归属清晰 |
| 7 | When（什么时候） | 2.3 版本规划中有明确的版本划分和优先级 |
| 8 | How（怎么做） | 2.2 产品整体流程、3.x 功能详述中操作步骤清晰，异常路径已覆盖 |
| 9 | How Much（多大程度） | 4.x 非功能需求中有性能、安全、数据量级等具体指标或说明 |
| 10 | 用户故事 | 每个故事有价值陈述、前后置条件、Happy Path、异常处理、验收标准 |
| 11 | 异常流程 | 每个功能的异常分支已覆盖（至少：校验失败、权限异常、网络异常、数据异常） |
| 12 | 数据字典 | 字段名、类型、必填、说明、示例值，缺一不可 |
| 13 | 验收标准 | 每个核心功能有至少一条 GIVEN-WHEN-THEN 验收条件 |
| 14 | 待确认项 | 所有假设和信息缺口已标注并汇总在附录 |
| 15 | 无空话 | 没有"提升体验"、"优化性能"等无法执行的描述 |
| 16 | 版本记录 | 文档头部有版本号、日期、修订人、备注 |
| 17 | 门禁检查 | 单轮模式：所有信息已标注 [需复核] 或 [假设] |

若任一项未通过，修正后再输出。
