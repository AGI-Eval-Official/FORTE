# PRD 撰写规范与 5W2H 框架说明

本文件为 prd_drafting SKILL 提供各章节的详细撰写规范，供生成 PRD 时查阅。

---

## 一、5W2H 框架在 PRD 中的应用

5W2H 是需求完整性的检查工具。在需求澄清阶段用它发现信息缺口，在 PRD 生成阶段用它验证每个章节是否覆盖到位。

> 5W2H 各维度与 PRD 章节的映射关系见 SKILL.md「Checklist（输出前自检）」，此处不再重复映射，仅展开各维度的撰写要点和常见错误。

### Why — 为什么做

**撰写要点**：
- 背景必须说清楚"发生了什么变化"，让读者理解为什么现在要做这件事
- 目标必须区分业务目标（对公司的价值）和用户目标（对用户的价值）
- 目标需可量化或可验证，禁止写"提升用户体验"等空话
- 衡量指标需写明当前值、目标值、衡量方式

**反面示例**：
- "为了提升用户体验" → 太空泛，应改为"当前用户完成XX操作平均需要5步，目标缩减到3步以内"
- "满足业务需求" → 无信息量，应改为"运营团队每周需手动处理XX条数据，耗时约8小时，目标实现自动化处理"

### Who — 谁来用

**撰写要点**：
- 目标用户不等于角色。用户描述要包含使用场景和核心诉求，角色描述要包含权限边界
- 如果有多种角色，必须说清每种角色能做什么、不能做什么
- 权限边界模糊是 PRD 评审中最常被挑战的点，宁可多写一行不能做什么

**检查清单**：
- 所有角色是否都已列出？
- 每种角色的权限边界是否清晰？
- 不同角色看到的页面/数据范围是否有差异？如果有，是否已说明？

### What — 做什么

**撰写要点**：
- 需求描述必须包含"非目标"（不做什么），防止范围蔓延
- 功能清单是全部功能点的索引，每个功能点需标注优先级
- 使用 P0（必须做）/ P1（应该做）/ P2（可以后做）三级优先级

**非目标的写法**：
不是简单说"不做XX"，而是要说明"看似相关但本次不做的是XX，因为XX"。让读者理解为什么不做，避免评审时反复讨论。

### Where — 在哪做

**撰写要点**：
- 每个功能需说明入口路径（用户从哪里进来）
- 如果涉及多端（Web/App/小程序），需说明各端的差异
- 页面之间的跳转关系需说清楚

### When — 什么时候

**撰写要点**：
- 避免写死具体日期，使用相对时间（如"V1.0上线后2周"）
- MVP 版本需明确"最少做到什么程度就能上线验证"
- 版本之间的依赖关系需说明（V1.1 是否依赖 V1.0 的某些功能）

### How — 怎么做

**撰写要点**：
- 整体流程用文字逐步描述，每步包含用户动作和系统响应
- 功能详述使用用户故事格式，每个故事自包含（读一张卡片就能理解全部需求信息）
- Happy Path 必须配合异常路径，不能只写理想场景
- 状态流转、校验规则、计算逻辑等业务规则用文字显式写出

### How Much — 多大程度

**撰写要点**：
- 非功能需求容易被忽略，但对开发影响很大
- 每类至少写一条：安全合规、埋点统计、性能要求、系统集成
- 如果无法确定具体值（如响应时间），写出建议值并标注 [待确认]

## 二、PRD 章节目录与撰写规范

PRD 严格按以下五大章节组织，不得随意调整模块顺序：

```
一、概述（Why + Who：为什么做、谁来用）
  1.1 背景介绍
  1.2 产品概述
  1.3 目标用户
  1.4 产品目标（业务目标 + 用户目标 + 衡量指标）
  1.5 角色与权限
  1.6 名词说明
  1.7 文档信息（版本记录、阅读对象）

二、产品描述（What + When + Where：做什么、什么时候做、在哪做）
  2.1 需求描述（含非目标说明）
  2.2 产品整体流程（主流程 + 子流程文字描述）
  2.3 版本规划（分期计划与优先级）
  2.4 产品框架（功能模块与页面结构说明）
  2.5 功能清单（功能点列表，含优先级与状态）
  2.6 全局说明（全局交互规则、异常处理通则、列表/分页/空态/加载规则）

三、功能需求详述（How：怎么做）
  每个功能模块包含：用户故事、前置条件与后置条件、操作流程、
  异常与分支流程、业务规则、数据字典、验收标准

四、非功能需求（How Much：多大程度）
  4.1 安全与合规
  4.2 统计需求/埋点
  4.3 性能需求
  4.4 系统集成

五、附录
  5.1 验收标准汇总与测试要点
  5.2 待确认项清单
  5.3 参考资料
```

以下为各章节的详细撰写规范。

### 1.1 背景介绍

**结构**：当前状况 → 存在的问题/机会 → 为什么现在要做

**写法**：用3-5句话讲清楚背景，不需要长篇大论，但必须让第一次读到这个文档的人理解上下文。

**关键信息**：
- 这个功能/产品属于哪个业务线？
- 当前面临什么问题或看到什么机会？
- 是否有外部因素驱动（如竞品动作、政策变化、技术可行性变化）？

### 1.2 产品概述

**要求**：2-3句话概括"这次要做什么"，相当于电梯演讲。任何人读完这段话都应该能理解产品的核心意图。

### 1.3 目标用户

**要求**：不只列角色名称，需描述使用场景和核心诉求。使用表格格式：用户类型、使用场景、核心诉求。

### 1.4 产品目标

**要求**：
- 业务目标：对公司或团队的价值，需有可验证的衡量方式
- 用户目标：对目标用户的价值，需从用户视角描述
- 衡量指标：使用表格格式，包含指标名称、当前值、目标值、衡量方式

### 1.5 角色与权限

**要求**：每种角色用一行说清"能做什么"。如果角色之间存在权限继承关系（如管理员拥有编辑者的所有权限），需说明。

### 1.6 名词说明

**要求**：只列本文档中有歧义或非通用含义的术语。不需要解释人人都懂的词。

### 1.7 文档信息

**要求**：包含版本记录表（版本号、日期、修订人、修订内容）、阅读对象、关联文档链接。

### 2.1 需求描述

**要求**：
- "做什么"：列出本次需求包含的功能点
- "不做什么"（非目标）：说明看似相关但不在本次范围内的内容，附原因

### 2.2 产品整体流程

**要求**：
- 主流程用文字编号逐步描述，每步包含"用户做什么"和"系统做什么"
- 子流程单独描述，标注从主流程的哪一步分出
- 禁止只写"用户提交表单，系统处理"这种模糊描述

### 2.3 版本规划

**要求**：使用表格格式，包含版本号、功能范围、优先级、预计周期。MVP 版本需单独标注。

### 2.4 产品框架

**要求**：用文字描述功能模块划分、各模块之间的关系、页面结构。目的是让读者建立全局视图。

### 2.5 功能清单

**要求**：使用表格格式，每个功能点一行，包含编号、所属模块、功能描述、优先级、当前状态。这是整个 PRD 的功能索引。

### 2.6 全局说明

**要求**：统一定义跨功能的通用规则，避免在每个用户故事里重复写。包含：
- 加载态、空态、列表分页、排序的通用规则
- 网络异常、权限不足、会话过期、并发冲突的统一处理方式
- 全局操作反馈的通用方式（Toast/弹窗/页面跳转）

### 3.x 功能需求详述（用户故事）

撰写规则和模块清单见 SKILL.md「四、用户故事撰写规范」，填写深度参考见 `example-us.md`，此处不再重复。

### 4.x 非功能需求

**要求**：每个子节使用表格格式，逐条列出具体要求。无法确定的值标注 [待确认] 并给出建议值。

### 5.1 验收标准汇总

**要求**：从各用户故事中提取验收标准，按模块汇总，补充整体测试要点（如兼容性测试、压力测试等）。

### 5.2 待确认项清单

**要求**：从全文中提取所有 [假设] 和 [待确认]，按三档优先级汇总：必须确认（阻塞开发）、建议确认（影响完整度）、可后续补充。每条标注出处（对应章节编号）。

## 三、PRD 写作通用原则

> 以下原则对应 SKILL.md Checklist 第10项「无空话」检查，是撰写时的具体指导。

**可执行 > 漂亮**：每个描述都要具体到开发能直接干活，不写"提升用户体验"这种无法执行的话。

**具体 > 模糊**：所有数值、文案、规则都要写出具体值。"合理时间内"应改为"≤ 3秒内"；"适当提示"应改为"Toast提示'操作成功'，显示2秒后自动消失"。

**显式 > 隐式**：所有隐含的业务规则都要用文字写出来。不要假设开发"应该知道"。

**一致性**：文档中同一个概念使用同一个名称，在名词说明中统一定义。

**完整性**：每个功能不仅写 Happy Path，还必须覆盖异常路径、边界条件、空态处理。
