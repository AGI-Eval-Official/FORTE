# 用户故事填写示例

本文件提供用户故事各模块的填写深度参考，供 prd_drafting SKILL 在生成 PRD 时对齐预期格式和颗粒度。

---

## 示例一：US-01 批量任务创建

```markdown
#### US-01: 作为POC测试用户，我希望上传多张作业图片并配置批改参数，以便一次性发起批量批改任务。

**价值陈述**：
- **作为** POC测试用户
- **我希望** 在一个页面完成图片上传与标准答案录入
- **以便于** 用最少的操作触发批量批改并减少配置错误

**前置条件**：
用户已登录；系统已加载可用的批改模型列表。

**操作流程（Happy Path）**：
1. 用户从导航栏进入"批量任务"页面。
2. 在图片上传区选择 ≤20 张、单张 ≤10MB 的图片；文件列表实时展示，支持单项删除。
3. 在标准答案输入区按"序号. 单词"格式录入标准答案，系统即时校验序号连续性和英文字符合法性。
4. 用户选择批改模型（默认项）、是否启用"智能终审"、以及批改权重模板（从下拉框加载）。
5. 所有校验通过后，"开始批改"按钮激活；点击后创建任务，页面跳转到任务监控视图。

**异常与分支流程**：

| 异常场景 | 触发条件 | 系统行为 | 用户提示文案 | 是否保留已输入数据 |
|----------|---------|---------|-------------|------------------|
| 图片超限 | 单张体积 >10MB 或扩展名不在白名单 | 拒绝该文件上传 | "该文件不符合要求，请压缩后重试" | 是（其他已上传文件保留） |
| 答案校验失败 | 序号不连续或含非英文字符 | 在输入框下方显示错误，禁用提交按钮 | "第3行缺少序号" | 是 |
| 超出批量上限 | 尝试上传第21张图片 | 拒绝上传 | "单次批改最多20张" | 是（已上传列表不变） |
| 任务创建失败 | 服务端返回错误 | 弹窗提示，保留已填数据 | "任务创建失败，请稍后重试" | 是 |

**业务规则**：
1. 图片格式白名单：jpg、jpeg、png、bmp
2. 单次提交上限：20张图片，总体积 ≤150MB
3. 标准答案格式：每行一条，格式为"序号. 英文单词"，序号从1开始连续递增
4. 批改模型默认选中系统推荐模型，用户可切换

**数据字典**：

| 字段名 | 类型 | 是否必填 | 说明 | 示例值 |
|--------|------|---------|------|--------|
| images | File[] | 是 | 上传的图片文件列表，≤20张 | [hw_001.jpg, hw_002.jpg] |
| standard_answers | String | 是 | 标准答案文本，按行分隔 | "1. apple\n2. banana" |
| model_id | String | 是 | 批改模型标识 | "model_default_v2" |
| enable_review | Boolean | 否 | 是否启用智能终审，默认 false | true |
| weight_template_id | String | 否 | 批改权重模板标识 | "template_01" |

**验收标准**：

**场景1: 成功提交**
- **GIVEN** 我上传了10张图片并输入合法标准答案
- **WHEN** 我点击"开始批改"
- **THEN** 系统应跳转到监控页，显示任务状态为"处理中 0/10"

**场景2: 超出批量上限**
- **GIVEN** 我已上传20张图片
- **WHEN** 我尝试上传第21张图片
- **THEN** 上传被拒绝，提示"单次批改最多20张"，已上传列表保持不变

**后置条件**：
任务创建成功后，系统在后台开始异步批改处理；任务状态变为"处理中"；用户可在任务监控页查看进度。
```

## 示例二：US-02 用户角色管理

```markdown
#### US-02: 作为系统管理员，我希望新增和管理用户角色，以便控制不同人员对系统功能的访问权限。

**价值陈述**：
- **作为** 系统管理员
- **我希望** 在后台创建角色并分配权限
- **以便于** 精细化控制团队成员的操作范围，降低误操作风险

**前置条件**：
用户已登录且拥有"系统管理员"角色权限；系统权限树已加载（包含所有可分配的权限项）。

**操作流程（Happy Path）**：
1. 管理员进入"系统设置 → 角色管理"页面，看到现有角色列表。
2. 点击"新增角色"按钮，弹出角色创建表单。
3. 填写角色名称（必填，≤20字符）和角色描述（选填，≤100字符）。
4. 在权限树中勾选该角色需要拥有的权限项，支持按模块批量勾选。
5. 点击"保存"，系统校验通过后角色创建成功，返回角色列表页，新角色出现在列表中。

**异常与分支流程**：

| 异常场景 | 触发条件 | 系统行为 | 用户提示文案 | 是否保留已输入数据 |
|----------|---------|---------|-------------|------------------|
| 角色名重复 | 输入的角色名与已有角色名一致 | 保存时校验失败，表单不关闭 | "角色名称已存在，请修改后重试" | 是 |
| 角色名为空 | 未填写角色名称直接点击保存 | 在字段下方显示校验错误 | "角色名称不能为空" | 是 |
| 未选择任何权限 | 权限树无任何勾选即点击保存 | 弹窗确认 | "当前未选择任何权限，确认创建空权限角色？" | 是 |
| 删除已绑定用户的角色 | 尝试删除已有用户绑定的角色 | 阻止删除 | "该角色下有 X 位用户，请先转移用户后再删除" | 不适用 |

**业务规则**：
1. 系统预置"超级管理员"角色不可编辑、不可删除
2. 角色名称全局唯一，不区分大小写
3. 权限变更实时生效：修改角色权限后，该角色下所有用户的权限立即更新（下次请求时生效）
4. 角色删除为软删除，保留30天后物理清除

**数据字典**：

| 字段名 | 类型 | 是否必填 | 说明 | 示例值 |
|--------|------|---------|------|--------|
| role_name | String | 是 | 角色名称，≤20字符，全局唯一 | "内容编辑" |
| role_desc | String | 否 | 角色描述，≤100字符 | "负责内容审核与发布" |
| permissions | String[] | 否 | 权限ID列表 | ["perm_content_edit", "perm_content_publish"] |
| status | Enum | 是 | 角色状态：active / inactive / deleted | "active" |

**验收标准**：

**场景1: 成功创建角色**
- **GIVEN** 我在角色创建表单中填写了唯一的角色名并勾选了至少一项权限
- **WHEN** 我点击"保存"
- **THEN** 系统创建角色成功，返回角色列表，新角色显示在列表中

**场景2: 角色名重复**
- **GIVEN** 系统中已存在名为"内容编辑"的角色
- **WHEN** 我尝试创建一个同名角色并点击保存
- **THEN** 系统提示"角色名称已存在，请修改后重试"，表单保持打开状态

**场景3: 删除有用户的角色**
- **GIVEN** "运营专员"角色下绑定了3位用户
- **WHEN** 我点击删除该角色
- **THEN** 系统阻止删除并提示"该角色下有3位用户，请先转移用户后再删除"

**后置条件**：
角色创建成功后，该角色出现在"用户管理 → 分配角色"的下拉选项中，可被分配给具体用户。
```
