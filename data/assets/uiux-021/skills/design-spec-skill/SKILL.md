# 设计规范文档 Skill

用于生成可直接给设计和研发使用的 UI 设计规范文档。

## 输入信息

尽可能从用户请求中提取以下信息：
- 产品类型与业务场景
- 目标平台（移动端、小程序、H5、iOS、Android）
- 品牌调性（中性、年轻、专业等）
- 主题模式（浅色、深色、双主题）
- 已有主色或约束条件
- 输出格式（如 `md`、`html`、`docx`、`json`）
- 输出内容范围（如“只要色彩+字体”“不要组件示例”“仅输出 token 表”）

优先识别用户可指定参数：
- 色彩：主色、功能色、中性色、状态色、暗黑模式色
- 字体：字号梯度、字重、行高、最小正文字号
- 图标：尺寸档位、描边粗细、圆角风格
- 间距：基础步长与间距刻度
- 圆角与阴影：层级数量、具体数值
- 组件：按钮/输入框/卡片等尺寸与状态样式
- 单位体系：`dp`/`sp`/`pt`/`rpx`/`px`/`rem`
- 输出格式：`md`/`html`/`docx`/`json`
- 输出内容：章节范围、是否包含组件示例、是否包含 token 映射、是否输出参数来源说明

## 参数优先级（必须遵守）

参数合并按以下顺序执行：
1. 用户显式指定值（最高优先级）
2. 用户给出的范围或约束（如“字号不小于 14”）
3. 平台 rules 文件默认值
4. `shared-rules.md` 通用默认值（兜底）

规则：
- 用户已指定的字段不得被默认值覆盖。
- 用户仅指定部分字段时，只覆盖对应字段，其余用默认值补齐。
- 若用户值与平台建议冲突，优先保留用户值，并在输出中标注影响与换算说明。

## 平台路由

按以下优先级选择平台规则文件：
1. 用户明确提到 `iOS`、`苹果端`、`Apple` 时，读取 `references/ios-rules.md`。
2. 否则，用户明确提到 `Android`、`安卓` 时，读取 `references/android-rules.md`。
3. 否则，用户提到 `小程序`、`微信小程序`、`xiaochengxu`、`rpx` 时，读取 `references/mini-program-rules.md`。
4. 否则，用户提到 `H5`、`Web`、`网页`、`浏览器`、`rem` 时，读取 `references/h5-rules.md`。
5. 否则，用户提到 `移动端`、`App`、`手机端`、`mobile` 时，读取 `references/mobile-rules.md`。
6. 若用户未提供平台信息，默认读取 `references/mobile-rules.md`。

始终读取 `references/shared-rules.md` + 1 个平台规则文件。
除非用户明确要求多端对比，否则不要混用多个平台规则文件。

## 执行流程

1. 识别平台并完成路由。
2. 提取用户参数与约束，形成参数清单。
3. 读取 `shared-rules.md` 与目标平台 rules 文件。
4. 按“用户优先”策略合并得到最终参数。
5. 生成含具体数值的设计 token 章节。
6. 生成组件规范示例，包含状态与尺寸。
7. 使用共享质检清单做一致性检查。
8. 按目标输出格式生成最终文档。

## 输出格式策略

- 用户明确指定输出格式时，按用户指定格式输出。
- 用户未指定输出格式时，默认输出 `md`。
- 若请求格式为 `docx`、`html`、`json`：
  - 结构章节保持一致；
  - `docx` 使用标题层级与表格组织；
  - `html` 输出语义化标题与表格；
  - `json` 输出结构化字段（包含平台、tokens、components）。

## 输出内容策略

- 用户明确指定输出内容时，严格按用户指定的章节范围输出。
- 用户未指定输出内容时，按默认流程输出完整章节。
- 用户指定“排除项”（如不含组件示例）时，必须从最终文档中剔除对应章节。
- 用户指定“仅输出项”（如仅 token 映射）时，只输出请求内容与必要上下文。

## 默认值注入（用户未指定时）

- 优先使用目标平台 rules 中的“未指定时默认参数表”。
- 平台文件未覆盖的字段，回退到 `shared-rules.md` 全局默认值。
- 默认值必须在文档的“参数来源说明”中标记为“默认补齐”。

## 输出要求

若用户未指定输出内容，默认输出完整章节：
1. 适用范围与目标平台
2. 色彩系统
3. 字体系统
4. 图标规范
5. 间距系统
6. 圆角与阴影规范
7. 组件示例
8. 面向研发的 token 命名映射

补充要求：
- 在文档开头增加“参数来源说明”（用户指定 / 默认补齐）。
- 输出数值时，单位必须与所选平台保持一致或给出换算说明。
- 用户显式指定的字号、色彩、间距、圆角等必须原样保留。
- 用户指定输出内容范围时，以用户指定范围为准。

## 多端请求处理

当用户明确要求多端时：
- 先给出一份共享基础规范
- 增加平台差异映射表（单位、交互、组件行为）
- 再分别补充各平台注意事项

## 质量标准

输出前必须确认：
- token 命名全篇一致
- 尺度梯度连续且可复用
- 可点击控件满足最小触控尺寸
- 正文字号可读性达标
- 关键状态齐全（默认、按下/悬停、禁用、错误/成功等）
- 用户指定参数未被默认值误覆盖
- 输出格式符合用户指定或默认 `md`
- 输出内容范围符合用户指定或默认完整流程
