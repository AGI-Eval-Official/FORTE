# 默认分析规则模板

> 本文件为通用规则参考模板。当用户未提供规则文档时，SKILL 将基于此模板执行分析。
> 用户可通过 instruction 或自定义规则文档覆盖本模板中的任何内容。

---

## 分析维度

| # | 维度名称 | 分析焦点 |
|---|----------|----------|
| 1 | 信息架构 | 导航结构、页面层级、信息分类方式、内容组织逻辑 |
| 2 | 交互设计 | 操作方式、手势支持、控件选用、交互反馈即时性 |
| 3 | 视觉呈现 | 排版布局、色彩运用、图标风格、视觉层次与一致性 |
| 4 | 任务流程 | 核心任务的操作步骤数、路径长短、中断恢复能力 |
| 5 | 反馈机制 | 操作反馈、错误提示、加载状态、空状态处理 |
| 6 | 容错与帮助 | 撤销能力、输入校验、帮助入口、新手引导 |

## 报告章节结构

```
一、逐维度对比分析    ← 每个维度含：A/B 表现描述、证据引用、评分、优势方判定、判断理由
二、综合评价          ← 双方各自的核心优势与主要不足归纳
三、设计建议          ← 针对双方的具体改进建议
```

## 评分标准（5 分制）

| 分值 | 含义 | 判断标准 |
|------|------|----------|
| 5 | 优秀 | 超出预期，有明显创新或行业领先实践 |
| 4 | 良好 | 符合主流最佳实践，小瑕疵不影响整体体验 |
| 3 | 一般 | 基本可用，存在明显改进空间 |
| 2 | 较差 | 存在影响体验的设计缺陷，用户可能感到困惑 |
| 1 | 很差 | 严重设计失误，导致任务失败或用户流失 |

> 同一维度内相对（双方差距）+ 绝对（行业标准）结合评判。

## 数量要求

- 综合评价：a_strengths / a_weaknesses / b_strengths / b_weaknesses 各 ≥ 2 条
- 设计建议：每款 App ≥ 2 条，每条须包含 issue / suggestion / related_dimension
