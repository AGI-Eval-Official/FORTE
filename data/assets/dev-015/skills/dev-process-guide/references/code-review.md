# Code Review 规范

## 一、Code Review 流程

### 1.1 提交 PR/MR 前的自查

开发者在提交 Code Review 前，必须完成以下自查：

- 代码已通过本地编译和单测
- 已删除调试代码（console.log、print、临时注释等）
- Commit message 符合规范
- PR/MR 描述清晰，包含：需求背景、改动说明、影响范围、测试情况
- 变更文件数量合理（建议单次 PR 不超过 500 行改动，超过应拆分）

### 1.2 Review 流程

1. 开发者创建 PR/MR，填写描述信息，指定 Reviewer
2. Reviewer 在 1 个工作日内完成 Review
3. Reviewer 提出修改意见，标注严重程度（Must Fix / Suggestion / Nit）
4. 开发者逐项回复和修改
5. 所有 Must Fix 项修复后，Reviewer 批准合入
6. 开发者合入代码到目标分支

## 二、Review 检查维度

### 2.1 功能正确性

- 代码逻辑是否正确实现了需求
- 边界条件是否处理（空值、越界、并发）
- 错误处理是否完善（异常捕获、错误码、错误信息）
- 数据校验是否充分（入参校验、业务规则校验）

### 2.2 代码质量

- 命名是否清晰、有意义
- 方法职责是否单一，是否过长
- 是否存在重复代码，是否可以抽取公共方法
- 是否遵循项目的编码规范和设计模式
- 注释是否充分且有意义

### 2.3 安全性

- 是否存在 SQL 注入风险
- 是否存在 XSS 风险
- 敏感数据是否加密处理
- 权限校验是否完善
- 日志中是否泄露敏感信息

### 2.4 性能

- 是否存在 N+1 查询问题
- 循环中是否有不必要的数据库/网络调用
- 大数据量场景是否考虑分页
- 是否有不必要的内存占用（如加载全量数据到内存）

### 2.5 可维护性

- 代码是否易于理解和修改
- 是否有充分的单元测试
- 配置项是否外部化（非硬编码）
- 是否考虑了向后兼容性

## 三、Review 意见分级

| 级别 | 标记 | 说明 | 是否必须修复 |
|------|------|------|-------------|
| 严重 | Must Fix | 功能缺陷、安全漏洞、数据风险 | 是 |
| 建议 | Suggestion | 代码优化、更好的实现方式 | 建议修复 |
| 细节 | Nit | 格式、命名等细微问题 | 可选 |

## 四、常见 Review 问题清单

- 缺少入参校验
- 异常被吞掉（catch 后无处理）
- 硬编码的魔法数字/字符串
- 方法过长（超过 50 行）
- 缺少必要的日志记录
- 未处理并发场景
- SQL 拼接存在注入风险
- 返回值未做空值判断
