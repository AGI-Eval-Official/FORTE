# 需求分析与技术方案设计规范

## 一、需求分析

### 1.1 需求理解

收到产品需求文档（PRD）后，开发者需要完成以下工作：

- **通读 PRD**：完整阅读需求文档，理解业务背景、用户场景和功能目标
- **识别功能边界**：明确本次需求的范围，区分必做项和可选项
- **梳理验收标准**：将 PRD 中的功能描述转化为可验证的验收条件
- **识别技术风险**：评估是否涉及新技术引入、性能瓶颈、数据迁移等风险点

### 1.2 需求分析产物

需求分析完成后应产出：

- 功能点清单（含优先级标注）
- 验收标准列表（每个功能点对应明确的验收条件）
- 技术风险评估（如有）
- 疑问与待确认项列表

## 二、技术方案设计

### 2.1 方案设计原则

- **最小改动原则**：在满足需求的前提下，尽量减少对现有系统的改动
- **可扩展性**：设计时考虑未来可能的扩展需求
- **一致性**：与现有系统的架构风格和设计模式保持一致
- **可测试性**：设计的模块应易于编写单元测试

### 2.2 技术方案内容（M/H 级必须）

技术方案文档应包含：

- **需求概述**：简要描述需求背景和目标
- **整体架构**：系统模块划分、模块间交互关系
- **详细设计**：核心模块的类设计、关键算法、状态流转
- **接口设计**：API 接口定义（请求/响应格式、错误码）
- **数据模型**：数据库表结构变更、字段说明
- **非功能性考虑**：性能、安全、兼容性等方面的设计

### 2.3 接口设计规范

- RESTful API 遵循标准 HTTP 方法语义（GET 查询、POST 创建、PUT 更新、DELETE 删除）
- 接口路径使用小写字母和连字符，如 `/api/v1/eval-tasks`
- 请求和响应使用 JSON 格式
- 统一错误响应格式，包含 code、message、data 字段
- 分页接口统一使用 page/pageSize 参数

### 2.4 方案评审

M/H 级需求的技术方案需要经过评审：

- 评审前将方案文档提前分享给评审人
- 评审重点关注：架构合理性、接口设计、数据模型、性能风险
- 评审意见需记录并逐项跟进
