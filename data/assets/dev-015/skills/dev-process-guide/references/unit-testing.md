# 单元测试规范

## 一、单测基本要求

### 1.1 覆盖率要求

- 核心业务逻辑：行覆盖率 ≥ 80%
- 新增代码：增量覆盖率 ≥ 80%
- 工具类/辅助方法：行覆盖率 ≥ 90%
- Controller 层：至少覆盖正常路径和主要异常路径

### 1.2 必须编写单测的场景

- 所有 Service 层的公共方法
- 核心业务逻辑（如状态流转、权限校验、数据计算）
- 工具类和辅助方法
- 数据转换和格式化逻辑
- 边界条件处理（空值、极值、非法输入）

### 1.3 可以不写单测的场景

- 简单的 getter/setter 方法
- 纯粹的数据传输对象（DTO）
- 框架自动生成的代码
- 仅做简单委托调用的方法

## 二、单测编写规范

### 2.1 测试用例命名

采用 `方法名_场景描述_预期结果` 的命名方式：

- `createEvalTask_withValidInput_shouldReturnTaskId`
- `createEvalTask_withNullName_shouldThrowException`
- `getEvalResult_withNonExistentId_shouldReturnNull`

### 2.2 测试结构（AAA 模式）

每个测试用例遵循 Arrange-Act-Assert 模式：

- **Arrange（准备）**：构造测试数据和依赖
- **Act（执行）**：调用被测方法
- **Assert（断言）**：验证执行结果

### 2.3 测试用例设计

每个被测方法应覆盖以下场景：

- **正常路径**：标准输入，预期正常输出
- **边界条件**：空值、空集合、极大/极小值、边界长度
- **异常路径**：非法输入、依赖服务异常、数据不存在
- **业务规则**：状态约束、权限校验、并发场景

### 2.4 Mock 使用规范

- 只 Mock 外部依赖（数据库、远程服务、文件系统），不 Mock 被测类自身的方法
- Mock 的行为应与真实行为一致（返回值类型、异常类型）
- 避免过度 Mock，如果需要 Mock 太多依赖，考虑重构被测代码
- 验证关键交互（如确认某个方法被调用了指定次数）

## 三、单测质量检查

### 3.1 检查清单

- [ ] 测试用例是否覆盖了所有公共方法
- [ ] 是否包含正常路径和异常路径的测试
- [ ] 边界条件是否被覆盖
- [ ] 断言是否充分（不仅检查返回值，还检查副作用）
- [ ] 测试是否独立（不依赖执行顺序，不依赖外部状态）
- [ ] 测试是否可重复执行（多次运行结果一致）
- [ ] 测试执行速度是否合理（单个测试 < 1 秒）

### 3.2 常见问题

- 只断言不抛异常，不验证返回值的正确性
- 测试数据硬编码且无意义（如 "test1"、"abc"）
- 测试之间存在依赖关系
- Mock 返回值与实际不符
- 缺少对边界条件的测试
