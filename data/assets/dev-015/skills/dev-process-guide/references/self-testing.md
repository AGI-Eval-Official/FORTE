# 自测验证规范

## 一、自测基本要求

### 1.1 自测时机

- 编码完成且单测通过后
- Code Review 修改完成后
- 本地启动服务后

### 1.2 自测范围

- **核心功能路径**：需求中定义的所有功能点
- **边界场景**：空数据、极值、特殊字符、超长输入
- **异常场景**：网络异常、服务不可用、数据不存在
- **兼容性**：与现有功能的兼容性验证
- **权限场景**：不同角色的访问权限验证

## 二、自测用例设计

### 2.1 用例设计原则

- 每个功能点至少一个正向用例和一个反向用例
- 用例应覆盖 PRD 中的所有验收标准
- 用例应包含前置条件、操作步骤、预期结果
- 用例之间应相互独立，不依赖执行顺序

### 2.2 用例模板

每个自测用例应包含：

- **用例编号**：唯一标识，如 TC-001
- **用例名称**：简要描述测试场景
- **前置条件**：执行用例前需要满足的条件
- **操作步骤**：详细的操作步骤描述
- **预期结果**：期望的系统响应和数据变化
- **实际结果**：执行后的实际情况
- **是否通过**：Pass / Fail

### 2.3 用例覆盖维度

针对 CRUD 类功能，自测用例应覆盖：

**创建（Create）：**
- 正常创建（所有必填字段）
- 缺少必填字段
- 字段值超出范围（过长、过短、非法字符）
- 重复创建（唯一性约束）

**查询（Read）：**
- 按 ID 查询存在的记录
- 按 ID 查询不存在的记录
- 列表查询（有数据、无数据）
- 分页查询（首页、末页、超出范围）
- 条件筛选查询

**更新（Update）：**
- 正常更新
- 更新不存在的记录
- 并发更新
- 部分字段更新

**删除（Delete）：**
- 正常删除
- 删除不存在的记录
- 删除有关联数据的记录

## 三、自测执行

### 3.1 执行规范

- 按用例编号顺序逐一执行
- 记录每个用例的实际结果
- 发现问题立即记录，标注严重程度
- 所有用例执行完毕后汇总结果

### 3.2 自测报告

自测完成后应产出自测报告，包含：

- 自测用例总数
- 通过/失败/阻塞数量
- 失败用例的问题描述和修复计划
- 自测结论（是否达到提测标准）

### 3.3 接口自测

对于后端接口，自测应包含：

- 使用实际的 HTTP 请求验证（如 curl、Postman）
- 验证请求参数校验逻辑
- 验证响应格式和字段完整性
- 验证错误码和错误信息的准确性
- 验证接口幂等性（如适用）
