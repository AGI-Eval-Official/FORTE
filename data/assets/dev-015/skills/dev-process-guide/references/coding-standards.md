# 编码开发规范

## 一、分支管理

### 1.1 分支策略

- **主分支（main/master）**：始终保持可发布状态，禁止直接提交
- **开发分支（feature/xxx）**：从主分支创建，功能开发完成后通过 PR/MR 合入
- **修复分支（hotfix/xxx）**：紧急 bug 修复，从主分支创建，修复后合入主分支
- 分支命名规范：`feature/模块名-功能简述`，如 `feature/eval-task-create`

### 1.2 提交规范

Commit message 遵循 Conventional Commits 规范：

- `feat: 新增评测任务创建功能`
- `fix: 修复评测结果分页查询异常`
- `refactor: 重构模型管理模块`
- `test: 补充评测集管理单元测试`
- `docs: 更新接口文档`

每次提交应保持原子性，一个 commit 只做一件事。

## 二、编码规范

### 2.1 通用规范

- 代码缩进统一使用空格（推荐 4 空格或 2 空格，项目内保持一致）
- 文件末尾保留一个空行
- 删除未使用的 import 和变量
- 避免硬编码，使用常量或配置项
- 方法长度不超过 50 行，超过应拆分
- 类的职责单一，遵循 SOLID 原则

### 2.2 命名规范

- 类名使用 PascalCase：`EvalTaskService`
- 方法名使用 camelCase：`createEvalTask`
- 常量使用 UPPER_SNAKE_CASE：`MAX_RETRY_COUNT`
- 变量名应具有描述性，避免单字母变量（循环变量除外）
- 布尔变量使用 is/has/can 前缀：`isCompleted`、`hasPermission`

### 2.3 注释规范

- 公共 API 必须有文档注释（描述功能、参数、返回值）
- 复杂业务逻辑需要添加行内注释说明意图
- 避免无意义的注释（如 `// 获取用户` 后面跟着 `getUser()`）
- TODO 注释需标注负责人和预计完成时间

### 2.4 错误处理

- 不要吞掉异常（catch 后不做任何处理）
- 使用具体的异常类型，避免捕获通用 Exception
- 业务异常和系统异常分开处理
- 错误信息应包含足够的上下文，便于排查

## 三、代码结构

### 3.1 项目分层

推荐采用分层架构：

- **Controller 层**：处理 HTTP 请求，参数校验，调用 Service
- **Service 层**：业务逻辑实现，事务管理
- **Repository/DAO 层**：数据访问，数据库操作
- **Model/Entity 层**：数据模型定义
- **DTO 层**：数据传输对象，用于层间数据传递

### 3.2 依赖管理

- 明确声明所有依赖，避免隐式依赖
- 定期更新依赖版本，关注安全漏洞
- 锁定依赖版本（使用 lock 文件）
