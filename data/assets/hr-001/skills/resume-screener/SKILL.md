# 简历筛选器

面向 HR 招聘场景的专业技能，用于系统化地将候选人简历与招聘 JD 进行比对筛选，输出结构化的筛选结论。整个流程分为三个阶段：JD 解析 → 简历分析 → 匹配输出，每个阶段独立可调用，便于在不同招聘场景中灵活复用。

> **⚠️ 最高优先级规则：优先以用户需求和用户规定要求为准。** 
> - 当用户明确提出筛选标准、匹配规则、通过条件等个性化要求时，必须以用户指令为最高优先级，本技能规则作为补充兜底。
> - 严格执行条件约束：用户指定“必须满足X”，则X为一票否决项；用户指定“X不重要”，则评估时完全排除X维度。
> - 用户自定义的分类方式、输出格式、评估维度等，均无条件覆盖并优先于本技能的默认设定。
> - 当用户的需求存在未覆盖的空白维度时，本技能的默认规则生效补位。

## 一、SOP 工作流程

### 第一步：读取并分析招聘 JD

逐份读取每个岗位的 JD 文档，完成以下四项任务：**提取结构化信息 → 划分要求等级 → 建立映射维度 → 确认模糊边界**。

#### 1.1 提取结构化信息

从 JD 文本中提取并结构化以下关键信息：

```
- 岗位名称
- 岗位所属部门/业务线
- 岗位职责（逐条列出，保留原始表述）
- 专业技能要求（逐条列出，区分"必须掌握"与"了解即可"）
- 加分项 / 优先条件（逐条列出）
- 工作经验要求（年限、行业领域、职级范围、管理幅度）
- 学历要求（最低学历、是否要求第一学历、优先专业方向）
- 其他条件（证书、语言能力、工作地点、出差频率、汇报关系等）
```

为每份 JD 分配唯一标识（如 `JD_001`），确保后续流程可追溯。

#### 1.2 划分要求等级

将提取出的所有要求按以下三个等级分类，**每个等级的判定直接影响后续匹配权重和通过/不通过结论**：

| 等级 | 定义 | 对匹配的影响 | 举例 |
|------|------|-------------|------|
| **硬性要求（Must-Have）** | 缺少则无法胜任该岗位的核心条件 | 任意一条不匹配 → 直接判定"不通过" | "3年以上BD经验"、"本科及以上学历"、"熟悉Python" |
| **软性要求（Should-Have）** | 重要但可通过培训/团队补充弥补的条件 | 影响匹配度和面试排序，不直接否决 | "有互联网行业经验优先"、"具备数据分析能力" |
| **加分项（Nice-to-Have）** | 有则加分、无则不扣分的条件 | 提升候选人优先级，但不影响通过判定 | "有海外工作经历"、"持有PMP证书" |

**等级划分原则**：
- JD 中出现"必须"、"要求"、"至少"、"须具备"、"硬性条件"等绝对化措辞 → 归为硬性要求
- JD 中出现"熟悉"、"了解"、"期望"、等常规描述，或无明显强调词的技能/经验 → 归为软性要求
- JD 中出现"优先"、"最好"、"首选"、"更佳"、"有...经验者优先"、"加分项"、"bonus"、"plus"等措辞 → 归为加分项
- 如果无法从措辞判断等级，默认归为软性要求，并在输出中标注"等级待确认"
- **用户明确指定的等级划分优先于上述原则**

#### 1.3 建立映射维度

为每条 JD 要求建立与简历信息的映射关系，明确"从简历的哪些位置寻找对应证据"：

| JD 要求维度 | 简历中对应的证据位置 | 证据强度判断 |
|------------|-------------------|------------|
| 专业技能 | 技能列表、项目经历中的技术栈描述、工作职责中的工具/方法提及 | 明确列出技能名称 > 项目中体现 > 仅提及关键词 |
| 工作经验 | 工作经历段（公司、职位、时长） | 同行业同岗位 > 同行业不同岗位 > 不同行业但职能相关 |
| 业绩能力 | 项目与业绩段的数据指标、成果描述 | 有量化数据 > 有定性描述 > 仅有职责罗列 |
| 学历背景 | 教育背景段 | 专业对口+学历达标 > 学历达标但专业不对口 > 学历不达标 |
| 管理能力 | 工作经历中的团队规模、汇报关系；项目中的协调角色 | 明确团队规模数据 > 提及"带领团队" > 仅有个人贡献 |
| 行业认知 | 工作经历中的行业/业务描述、项目中的业务领域 | 同细分行业 > 同大行业 > 跨行业但逻辑可迁移 |

#### 1.4 确认模糊边界

JD 中常见模糊表述需在解析阶段明确边界，**避免在匹配阶段自行猜测**：

| 模糊表述 | 需确认的边界 | 默认处理（未被用户覆盖时） |
|---------|------------|----------------------|
| "熟悉XX" | "熟悉"的程度：了解概念？使用过？精通？ | 默认=使用过并能独立完成相关任务 |
| "XX经验优先" | 最低多少年算"有经验"？ | 默认=1年以上相关经历 |
| "大型项目经验" | 多大规模算"大型"？ | 默认=跨团队/跨部门协作项目，或可量化的业务影响 |
| "互联网大厂" | 哪些公司算"大厂"？ | 默认=头部互联网公司（BAT/TMD/字节/快手等） |
| "良好的XX能力" | 如何衡量"良好"？ | 默认=有可展示的成果或项目证据 |

**关键原则**：JD 解析的完整性直接决定筛选质量。如果 JD 文本中有模糊表述且用户未定义边界，应在输出中标注"边界待确认，当前按默认标准处理"，而非自行猜测标准。**用户对模糊边界的定义优先于默认处理。**

#### 📌 示例：JD 解析结果

```
JD_001 · 外卖商户BD

【硬性要求】
1. 3年以上商务拓展或商户运营经验
2. 具备独立完成商户拓展全流程的能力（从触达到签约）
3. 本科及以上学历

【软性要求】
1. 有互联网/O2O行业BD经验优先
2. 具备数据分析能力，能基于数据调整策略
3. 良好的跨部门协作能力
4. 自驱力强，能适应快节奏工作

【加分项】
1. 有美团/饿了么等同平台工作经验
2. 有餐饮行业资源
3. 有团队管理经验

【映射维度】
- "3年BD经验" → 简历工作经历段：职位含BD/拓展/商务 + 累计时长≥3年
- "独立完成全流程" → 项目经历段：是否有从0到1的拓展项目，含触达-签约完整链路
- "数据分析能力" → 技能段或项目段：是否提及数据工具（Excel/SQL/BI）+ 是否有数据驱动决策的描述
- "跨部门协作" → 项目经历段：是否涉及多团队协调推进的描述

【模糊边界确认】
- "3年BD经验"：3年指累计相关岗位时长，非单段经历；BD/拓展/商户运营均计入
- "数据分析能力"：默认=至少能使用Excel/SQL做基础分析，有实际案例支撑
- "互联网大厂"：用户未指定，按默认（BAT/TMD/字节/快手等）处理
```

### 第二步：识别与分析简历内容

逐份读取每个岗位下的候选人简历，完成以下三项任务：**提取简历关键信息 → 按 JD 映射维度对标 → 标注信息完整度与缺失**。

#### 2.1 提取简历关键信息

从每份简历中提取以下结构化信息：

```
- 候选人姓名
- 联系方式（如有）
- 工作经历（每段：公司名称、职位、起止时间、在职时长、核心职责、核心成果）
- 技能与专长（明确列出的技能 + 从项目经历中推断的技能）
- 教育背景（每段：学校、专业、学位、起止时间）
- 证书与荣誉
- 项目与业绩（优先关注可量化的结果：增长率、完成量、规模等）
- 其他亮点（行业资源、特殊经历等简历中提及但未被上述维度覆盖的信息）
```

为每份简历标注来源文件名，确保可追溯。

#### 2.2 按 JD 映射维度对标

将提取的简历信息与第一步建立的映射维度逐条对标，**重点关注证据的充分性和可靠性**：

**证据充分性分层**：

| 证据等级 | 定义 | 示例 |
|---------|------|------|
| **强证据** | 简历中有明确的、可验证的事实直接对应 JD 要求 | JD 要求"3年BD经验"，简历显示"4年商户拓展经历，负责XX区域" |
| **中证据** | 简历中有相关描述，但需要进一步推断或验证 | JD 要求"数据分析能力"，简历项目提及"通过数据分析优化策略"但未列出具体工具 |
| **弱证据** | 简历中间接提及，关联性需主观判断 | JD 要求"跨部门协作"，简历仅写"协调各方资源推进项目" |
| **无证据** | 简历中完全未提及相关信息 | JD 要求"SQL能力"，简历无任何数据相关描述 |

**对标时的关键规则**：
1. **只看简历写了什么，不推测简历没写的内容** — 如果简历未提及某项技能/经历，即使候选人可能具备，也按"无证据"处理
2. **量化数据优于定性描述** — "商户增长200%"的证明力强于"显著提升商户数量"
3. **近期经历权重大于早期经历** — 最近2年的工作内容对岗位匹配的参考价值最高
4. **同行业/同岗位的直接经验权重大于间接经验** — 在美团做BD > 在传统行业做销售
5. **项目成果权重大于职责罗列** — "主导XX项目，达成YY结果" > "负责XX相关工作"

#### 2.3 标注信息完整度与缺失

对每份简历标注信息完整度，识别可能影响匹配判定的缺失：

| 标注类型 | 定义 | 后续处理 |
|---------|------|---------|
| **信息完整** | 所有 JD 映射维度均有对应证据 | 正常进入匹配阶段 |
| **部分缺失** | 部分映射维度在简历中找不到对应证据 | 在匹配判定中标注"简历未提供相关信息"，按"无证据"处理，列入【风险与疑点】 |
| **关键缺失** | 硬性要求对应的映射维度缺失 | 该条判定为"不匹配"，但标注"可能因简历未体现而非实际不具备"，列入【追问清单】 |
| **信息矛盾** | 简历不同位置的信息存在冲突 | 以更详细/更近期的信息为准，矛盾点列入【风险与疑点】 |

> **⚠️ 核心提醒**：简历中"未提及"不等于"不具备"。对于硬性要求的缺失，应区分"简历明确表明不符合"与"简历未提供相关信息"两种情况——前者可直接判不匹配，后者需在风险中标注并建议面试确认。

#### 📌 示例：简历分析与 JD 标对

```
候选人：张三 | 来源：张三_简历.pdf

【工作经历】
- 美团到店事业部 | 商务拓展经理 | 2022.03-2025.01（2年10个月）
```

### 第三步：JD-简历匹配与筛选输出

将每份简历的内容与对应 JD 的各项要求逐条比对，输出结构化筛选结果。

**匹配判定逻辑**：

| 判定 | 定义 | 判定条件 |
|------|------|---------|
| **匹配** | 简历提供了明确证据满足该条要求 | 存在强证据或中证据，且满足 JD 该条的硬性标准 |
| **部分匹配** | 简历有相关经验或技能，但未完全覆盖该条要求 | 技能栈/业务经验仅部分重合（注：硬性指标如学历、核心年限不达标） |
| **不匹配** | 简历缺乏任何相关证据，或明确不符合 | 包含两种情况：（1）明确不符合（如要求统招本科，实际为大专）；（2） 简历未体现（无任何相关证据） |

每条判定都必须引用简历中的具体内容作为证据（项目名、数据指标、职位名称等），避免主观臆断。

**通过/不通过判定规则**：
  - 任意硬性要求“明确不符合” → 不通过（一票否决）
  - 硬性要求存在“简历未体现” → 待定/存疑（默认暂不通过，提示需人工进一步核查）
  - 硬性要求全部“匹配” → 初筛通过
  - 面试优先级排序：在硬性要求通过的前提下，加分项越多、软性要求匹配度越高，推荐优先级越高；若软性要求多为“部分匹配”或“不匹配”，则降低优先级
- **用户自定义通过规则优先于上述默认规则**

## 二、输出格式规范

优先满足用户的明确输出要求，若用户无明确输出要求，则按照以下默认格式输出：
默认格式字段名称必须保持一致，不得增删改名。

### 通过筛选的候选人（7 个字段）

```
【简历筛选结论】通过
【JD 满足度】满足 X/Y 条硬性要求，满足 X/Y 条软性要求，满足 X/Y 条加分项（全部满足 / 大部分满足 / 部分满足）
【JD 匹配条目】（逐条列出：匹配/部分匹配/不匹配 + 证据等级 + 简历证据）
【亮点证据】（来自简历的具体经历/项目/结果，突出与岗位最相关的内容）
【风险与疑点】（需要核实的点，区分"明确不符合"与"简历未体现"，避免主观判断）
【追问清单】（面试要追问的 3-8 个问题，优先覆盖"部分匹配"和"简历未体现"的硬性要求）
【面试优先级与建议】（基于满足程度排序 + 建议面试轮次/侧重点）
```

**【JD 满足度】说明**：
- 格式为"满足 X/Y 条硬性要求，满足 X/Y 条软性要求，满足 X/Y 条加分项"
- 附加总体判断：全部满足（所有硬性+软性均匹配）/ 大部分满足（硬性全部满足，软性满足≥60%）/ 部分满足（硬性全部满足，软性满足<60%）
- 该字段直接决定面试排序：全部满足 > 大部分满足 > 部分满足

**【面试优先级与建议】说明**：
- 面试优先级分三档：**优先安排**（全部满足）/ **正常安排**（大部分满足）/ **备选安排**（部分满足）
- 同档内部排序依据：加分项满足数 > 软性要求满足数 > 亮点证据强度
- 格式示例："优先安排（P1） | 一面深挖数据分析能力验证；二面验证跨部门协作与自驱力"

### 未通过筛选的候选人（4 个字段）

```
【简历筛选结论】不通过 / 待补充信息
【JD 匹配条目】（逐条列出：匹配/部分匹配/不匹配 + 证据等级 + 简历证据）
【亮点证据】（来自简历的具体经历/项目/结果）
【风险与疑点】（需要核实的点，区分"明确不符合"与"简历未体现但可能具备"）
```

**"待补充信息"判定**：当不通过的原因为"简历未体现"而非"明确不符合"时，结论为"待补充信息"，表示如果候选人补充相关信息后可能通过筛选。

### 汇总行

每个输出文件的第一行必须是汇总统计：

```
通过xx人（优先安排xx人，正常安排xx人，备选安排xx人），不通过xx人，待补充信息xx人。
```

### 面试排序规则

通过筛选的候选人按以下规则确定面试先后顺序：

1. **第一排序键**：JD 满足度等级 — 全部满足 > 大部分满足 > 部分满足
2. **第二排序键**：加分项满足数量 — 多 > 少
3. **第三排序键**：软性要求满足比例 — 高 > 低
4. **第四排序键**：亮点证据强度 — 有量化数据 > 有定性描述 > 仅有职责罗列
5. **用户自定义排序规则优先于上述默认规则**

## 三、可复用指南

本技能设计为可复用的构建模块，适用于不同招聘场景，只需做最小化配置调整。

### 3.1 适配新岗位

1. **替换 JD 文件** — 将输入目录指向新的 JD 文档集，按第一步流程重新解析。
2. **调整要求等级** — 根据岗位特点，重新划分硬性要求/软性要求/加分项，确保关键条件不会被遗漏或降级。
3. **调整匹配权重** — 对于有"必须项"与"加分项"之分的岗位，可将核心要求设为一票否决项：硬性要求不匹配 → 直接不通过。
4. **自定义输出字段** — 基础的 4 字段（不通过）/ 7 字段（通过）结构应保持一致以确保下游流程可解析，但可在现有字段内增加内容细节。

### 3.2 多岗位批量筛选

对目录下所有 JD 批量筛选简历时，每个岗位生成一份输出文件。需注意：
- 不同岗位的 JD 解析结果独立维护，不可混用
- 同一候选人可同时出现在多个岗位的筛选结果中，各自独立判定
- 汇总行按岗位分别统计

## 四、核心原则

1. **优先以用户需求为准** — 用户明确规定的筛选标准、匹配规则、通过条件、输出格式等，优先于本技能的所有默认设定。本 SOP 仅在用户未指定时提供兜底逻辑。

2. **证据驱动判断** — 每条匹配/不匹配的判定都必须引用简历中的具体内容，标注证据等级，避免"看起来合适"等主观判断。

3. **结构化一致性** — 严格使用规定的字段名称和格式输出，确保下游流程（面试安排、候选人追踪）可以可靠解析结果。

4. **阶段独立可调用** — JD 解析、简历分析、匹配筛选三个阶段相互独立。如果用户只需要某个阶段（如"帮我提取这个 JD 的关键要求"），则只执行该阶段，不必走完整个流程。

5. **区分"不具备"与"未体现"** — 简历中未提及某项能力，不等于候选人不具备该能力。对于硬性要求的缺失，需区分"简历明确表明不符合"与"简历未提供相关信息"，前者直接判定不匹配，后者标注为风险并建议面试确认。

6. **模糊信息兜底** — 当 JD 或简历缺乏明确信息时，按默认标准处理并标注"待确认"，而非自行猜测。任何默认处理的边界都应透明告知用户。

7. **满足度驱动排序** — 通过筛选的候选人按 JD 满足程度排序，满足条件越多、证据越充分的候选人优先安排面试，确保有限的面试资源优先投入最匹配的候选人。
